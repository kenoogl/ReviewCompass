## 2. 7 機能の一覧と責務

| 機能名 | 責務概要 | 由来 |

| **workflow-management** | ワークフロー管理。所定手続き、reopen、整合ゲート、機能依存マップ管理 | 前システム継承（旧 implementation-governance 改称、§5.4 軽量化方針で大幅再設計） |

## 3. 機能依存関係（計画書 §3.1 から転記）

```yaml
features:
  foundation:
    depends_on: []
  runtime:
    depends_on: [foundation]
  evaluation:
    depends_on: [foundation, runtime]
  analysis:
    depends_on: [foundation, evaluation]
  workflow-management:
    depends_on: [foundation, runtime, evaluation, analysis]
  self-improvement:
    depends_on: [foundation, workflow-management]
  conformance-evaluation:
    depends_on:
      foundation: hard
      runtime: review
      evaluation: review
      workflow-management: review

phase_order:
  - foundation
  - runtime
  - evaluation
  - analysis
  - workflow-management
  - self-improvement
  - conformance-evaluation
```

依存関係の詳細：

- `hard`：強い依存（必須）
- `review`：レビュー依存（参照のみ）
- conformance-evaluation の self-improvement への出力は依存ではなく外部参照（A-008 で確定、機能間波及所見対処済み）

## 7. intent 更新後の再確認（2026-06-08）

2026-06-08 に intent 文書へ、レビュー収集処理を事前設定の写像にしない意図が追加された。

この意図は、機能分割上は新機能を追加せず、既存の責務境界で扱う。

- 受け皿がない場合は新 feature を作成するが、今回は既存 feature の責務境界で受けられるため新 feature は作成しない。
- 受け皿がある場合でも、既存 feature を一律に reopen するのではなく、実装上の所有を軸に、直接影響・間接確認・影響なしに分ける。intent 文書内の挿入箇所は、intent の性質上、feature 所属判定の主根拠にしない。
- foundation は、固定パターン依存の除外、実行メタデータ、証拠スキーマ、検証成果物分離の共有契約を直接受けるため、reopen 対象とする。
- runtime は、実際の大規模言語モデルの判断を用いてレビューを実行する責務を直接受けるため、reopen 対象とする。
- evaluation は、収集結果の妥当性と無効化を直接受けるため、reopen 対象とする。
- conformance-evaluation は、実装コードからの推定、既存上流文書との比較、契約所有候補、仕様更新草案を実装上の所有責務として持つため、reopen 対象とする。
- analysis は、収集結果を後段で構造化して扱い、runtime／evaluation の実行判断を上書きしないため、間接確認のみとする。
- workflow-management は、この再確認のように上流変更を下流段階へ順に反映する流れを管理するため、手続き整合の間接確認のみとする。
- self-improvement は、観測事実と証跡から改善提案を生成する後段であるため、間接確認のみとする。

この補正は、`.reviewcompass/specs/_cross_feature/reviews/2026-06-09-feature-impact-reopen-review-run/review_summary.md` の3役レビュー結果を踏まえつつ、実装上の所有を主軸に再検討したもの。direct impact feature の再確認で既存責務を超えると判明した場合は、feature-partitioning に戻して新 feature 要否を再判定する。

したがって、7 機能の構成と依存関係は変更しない。

## 8. operation contract intent 追記後の再確認（2026-06-19）

2026-06-19 に intent 文書へ、LLM が手順や不可逆操作の承認を暗黙に代行しないこと、`next --json` を唯一 action selector として扱うこと、operation contract / read-only preflight / 承認ゲート / 状態スナップショット / 有効プロンプトを構造化することが追記された。

この追記は、機能分割上は新機能を追加せず、既存の workflow-management の責務境界で扱う。理由は次の通り。

- operation contract、唯一 action selector、承認ゲート、side track、状態スナップショット、有効プロンプトは、いずれもワークフローの現在位置、次操作、停止点、復帰先を管理する責務である。
- これらは既存の workflow-management が持つ「所定手続き、reopen、整合ゲート、機能依存マップ管理」を具体化する内容であり、新しい独立 feature ではない。
- foundation は共有契約やスキーマを所有するが、今回の追記は共有レビュー抽象や基盤スキーマの新設ではない。
- runtime、evaluation、analysis、self-improvement、conformance-evaluation は workflow 状態や operation contract を参照し得るが、今回の所有責務は workflow-management に閉じる。

したがって、7 機能の構成、依存関係、名称、順序は変更しない。

この節は、2026-06-19 の利用者判断に基づき、当該 intent 追記が機能分割変更を要しないことを feature-partitioning 側で明示するための再確認記録である。reopen 手続きを無効化するための例外ではなく、feature-partitioning の候補提案として「既存分割で足りる」と確認した記録として扱う。
