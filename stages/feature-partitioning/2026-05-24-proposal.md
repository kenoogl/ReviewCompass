## 2. 7 機能の一覧と責務

| 機能名 | 責務概要 | 由来 |
|---|---|---|
| **foundation** | 4 段論理契約、3 役抽象名、5 スキーマ集合、20 必須メタデータ項目、4 状態軸等の基盤定義 | 前システム継承 |
| **runtime** | 実行時機能。3 役レビューの実行経路、API 呼び出し、ファイル遮断 | 前システム継承 |
| **evaluation** | 評価機能。5 段パイプライン、4 状態区分、3 階層 2 層指標、3 経路別差分出力 | 前システム継承 |
| **analysis** | 分析機能。10 メトリクスカテゴリ、4 出力先、レビュー収束過程の可視化 | 前システム継承（旧 paper-interface 改称） |
| **workflow-management** | ワークフロー管理。所定手続き、reopen、整合ゲート、機能依存マップ管理 | 前システム継承（旧 implementation-governance 改称、§5.4 軽量化方針で大幅再設計） |
| **self-improvement** | 自己改善機能（workflow 層のみ）。規律の昇格・退避・統廃合の提案権 | 前システム継承（§5.16 全面書き直し） |
| **conformance-evaluation** | 規律遵守検査機能。12 criteria による適合性評価 | 新規追加（§5.10、セッション 18 確定） |

## 3. 機能依存関係（計画書 §3.1 から転記）

```yaml
features:
  foundation:
    depends_on: []
  runtime:
    depends_on: [foundation]
  evaluation:
    depends_on: [foundation, runtime]
  analysis:
    depends_on: [foundation, evaluation]
  workflow-management:
    depends_on: [foundation, runtime, evaluation, analysis]
  self-improvement:
    depends_on: [foundation, workflow-management]
  conformance-evaluation:
    depends_on:
      foundation: hard
      runtime: review
      evaluation: review
      workflow-management: review

phase_order:
  - foundation
  - runtime
  - evaluation
  - analysis
  - workflow-management
  - self-improvement
  - conformance-evaluation
```

依存関係の詳細：

- `hard`：強い依存（必須）
- `review`：レビュー依存（参照のみ）
- conformance-evaluation の self-improvement への出力は依存ではなく外部参照（A-008 で確定、機能間波及所見対処済み）

## 4. 機能分割の根拠

- 前システムで実証された 6 機能の責務境界をほぼそのまま継承

- conformance-evaluation の追加は、規律遵守の機械検査を独立した機能として明示するため（§5.10）

## 7. intent 更新後の再確認（2026-06-08）

2026-06-08 に intent 文書へ、レビュー収集処理を事前設定の写像にしない意図が追加された。

この意図は、機能分割上は新機能を追加せず、既存の責務境界で扱う。

- 受け皿がない場合は新 feature を作成するが、今回は既存 7 feature の責務境界で受けられるため新 feature は作成しない。
- 受け皿があるため、該当する既存 feature を reopen 対象として扱う。今回の intent はレビュー収集、評価、仕様・実装乖離確認、workflow 再展開、共有契約、分析、改善にまたがるため、7 feature すべてを reopen 対象とする。
- runtime は、実際の大規模言語モデルの判断を用いてレビューを実行する責務を持つ。
- evaluation は、収集結果の妥当性と無効化を扱う責務を持つ。
- conformance-evaluation は、実装・仕様・意図のずれを検査する責務を持つ。
- workflow-management は、この再確認のように上流変更を下流段階へ順に反映する流れを管理する責務を持つ。
- foundation は、固定パターン依存の除外、実行メタデータ、証拠スキーマ、検証成果物分離の共有契約を持つ。
- analysis は、収集結果を後段で構造化して扱い、runtime／evaluation の実行判断を上書きしない責務を持つ。
- self-improvement は、観測事実と証跡から改善提案を生成し、場当たり的な規則追加にしない責務を持つ。

したがって、7 機能の構成と依存関係は変更しない。
