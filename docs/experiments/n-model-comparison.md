# N モデル比較実験ノート — 人間代役機構（計画書 §5.12）の検証

最終更新：2026-05-27（セッション 31、初版作成、第 1 回ケースの実験実施前）

本ノートは、ReviewCompass の人間代役機構（計画書 §5.12）の権限範囲チューニングに使う実証データを取得する実験の記録である。論文のエビデンスとして使えるよう、共通フレームワーク（§1〜§3）と個別ケース（§4〜§6、将来は §7 以降に追加）の 2 層構成で記述する。

## 1. 実験意図と背景（共通）

### 1.1 検証目的

計画書 §5.12 で定めた **人間代役機構** は、ReviewCompass の利用者（人本人）の判断を外部モデルに代行させて自動実行の継続性を上げる機構である。§5.12.4 で代役の権限範囲を「軽い判断は代役、重い判断は本人」と線引きしているが、この線引きの妥当性は実証データで裏付けられていない。

本実験は、複数のモデルを判定者として並べ、同じ判定対象に対する判断のばらつきを観察することで、§5.12.4 の線引きを実証データで裏付ける（または見直しの根拠を作る）。

### 1.2 副次的観察

同一モデル（Sonnet 4.6）の **CLI 経路**（Claude Code 内の Agent ツール経由）と **API 経路**（providers.py の直接 API 呼び出し経由）で判断が変わるかを観察する。両者は同じモデルだが、Claude Code が経路上で何らかの干渉をしている可能性がある（実際に環境変数の干渉が確認されている、§3.1 参照）ため、判断結果にも差が出る可能性がある。

### 1.3 ケース選定の方針

実 ReviewCompass 開発で発生した triad-review（3 役レビュー）の結果を実データとして使用する。仮想的な題材ではなく、実際の機能仕様文書のレビュー所見を判定対象にする。

機能・段に特化せず、複数ケースを段階的に蓄積する。第 1 回は foundation 機能の tasks 段で発生した triad-review の結果（must-fix 10 件 + should-fix 7 件 = 17 件）を対象にする。将来は他機能（runtime／evaluation／analysis 等）や他段（design／implementation 等）の結果も同じ枠組みで扱う。

### 2.2 共通プロンプト雛形の構成

判定者全員に同じプロンプトを投げる。プロンプトの構成：

1. **役の立ち位置**：「最終判断者」として依頼されたことを冒頭で明示
2. **文脈説明**：ReviewCompass の仕様駆動レビュー方式、3 役レビュー（提案役・反論役・判定役）の説明
3. **判定対象**：所見番号 ＋ 事実 ＋ 候補案 ＋ 深掘り
   - **「推奨」は判定者に見せない**（公平性確保、再オープン履歴は §4.4 に記録）
4. **判定の選択肢**：採用：案 1 ／ 採用：案 2 ／ 別案を提示 ／ 深掘り要求 の 4 値
5. **対話形式**：マルチターン（最大 5 ターン、§5.12.3 既定値）
6. **出力形式**：YAML 9 項目（§2.4 で詳述）

### 2.3 進め方の標準フロー

**案 B：論点 1 で予備実験 → プロンプト調整 → 残り論点で本実験**

1. 第 1 論点について 8 者全員に判定を依頼（予備実験）
2. 結果を点検：判定形式の妥当性、プロンプトの曖昧さ、出力構造の安定性
3. 必要ならプロンプトを修正、第 1 論点を再投入
4. 修正版で第 2 論点以降を一括実行（本実験）
5. 結果を集約、§5 と §6 に追記

### 2.4 結果集約のフォーマット

出力 YAML の必須 9 項目：

| 項目 | 内容 | 用途 |
|---|---|---|
| `decision` | 採用：案 1 ／ 採用：案 2 ／ 別案を提示 ／ 深掘り要求 | 主比較項目 |
| `rationale` | 判断の理由（3〜10 行） | 質的分析 |
| `alternative_proposal` | 別案の内容（decision = 別案を提示 のときのみ） | 新案発見 |
| `confidence` | 信頼度 0〜1 | 信頼度比較 |
| `turns_used` | 実際に使ったターン数 | 対話深度の指標 |
| `uncertainty_factors` | 判断に残る不確実性の要因（複数列挙） | 信頼度の根拠可視化 |
| `assumed_context` | 判断にあたって暗黙に置いた前提（複数列挙） | 前提条件の食い違い検出 |
| `case_scores` | 各候補案ごとの 0〜10 評価 | 4 値判定では見えない迷いの強さ |
| `comment_to_human` | 人本人への伝言（自由記述） | エスカレーション宣言の救出経路 |

結果は **判定者（8 者）× 論点数（M 件）** の表として集約する。第 1 回ケースでは 8 × 17 = 136 セル。

分析項目：

1. **人本人との一致率**（モデル別）
2. **経路差**（Sonnet CLI vs API）の観察
3. **モデル間の判断パターン差**（一致する判断と乖離する判断の傾向）
4. **重大度別の一致率傾向**（must-fix vs should-fix）
5. **§5.12.4 権限範囲チューニングへの示唆**

### 2.5 §0.3 起草者と判定者の分離規律との関係

triad-review では起草者と判定者を分離する規律（§0.3、計画書 §5.4）がある。本実験では：

- **Opus 4.7（本人）**：レビュー記録の判定役として既に判定済み。実験では既出推奨案を流用 → 規律違反なし（同じ役の再判定なので妥当）
- **Sonnet 4.6（CLI／API）**：レビュー記録の主役として参加済み。実験で「修正案の採用判定」をさせると、主役の所見を提案した者が判定者になる構図 → 厳密には §0.3 違反の懸念あり

本実験の目的は「人間代役機構の検証」（§5.12 由来）で、5 者（本実験では 7 モデル）の判断のばらつきを観察することなので、§0.3 違反の構図を含めて観察対象に含める方針（セッション 31、利用者明示承認「あ」）。

### 3.1 API キーの問題と解決

#### 確認手順と結果（セッション 31、2026-05-27）

各 API キーの環境変数を確認した（値は表示せず、設定の有無と文字数のみ）：

| 環境変数名 | .zshrc の記述 | zsh 経由で取得 | Claude Code 直下で取得 | 干渉の有無 |
|---|---|---|---|---|
| `OPENAI_API_KEY` | 行 135 | 164 文字 | 164 文字 | **干渉なし** |
| `ANTHROPIC_API_KEY` | 行 138 | 108 文字 | 0 文字（空に上書き） | **干渉あり** |
| `GEMINI_API_KEY` | 行 141 | 39 文字 | 0 文字（空に上書き） | **干渉あり** |
| `ANTHROPIC_BASE_URL` | 記述なし | `https://api.anthropic.com`（25 文字） | 同左 | Claude Code が注入、公式エンドポイント |

#### 原因の推定

Claude Code（Anthropic 社製アプリ）は子プロセスを起動するときに、ある種の環境変数を上書きしている。具体的には：

- `ANTHROPIC_API_KEY` を空文字列で上書き（利用者の値を子プロセスに渡さない）
- `ANTHROPIC_BASE_URL` を Anthropic 社の公式エンドポイント（`https://api.anthropic.com`）に注入
- `GEMINI_API_KEY` も空文字列で上書き（理由は不明、推測：Anthropic が競合プロバイダーの認証情報も同等に扱っている可能性）
- `OPENAI_API_KEY` には干渉していない

これは Claude Code 本体が Anthropic 社の認証情報を OAuth で管理している設計の副作用と推測される。

#### 解決方法：案 A（zsh 経由）

`zsh -c 'source ~/.zshrc && python3 ...'` の形で Python を起動すると、`.zshrc` 経由で 3 種類すべての API キーが取れる。動作確認済み：

```
OPENAI_API_KEY 長さ=164
ANTHROPIC_API_KEY 長さ=108
GEMINI_API_KEY 長さ=39
```

`ANTHROPIC_BASE_URL` は公式エンドポイントを指しているため、書き換えなし。Anthropic 社の正規サーバーに直接届く。

#### 採用しなかった案

- **案 B**：`.env` ファイル（環境変数をファイルから読み込む仕組み）を作成して `run_role.py` に読み込ませる → コードに変更が必要、テストも追加が必要、不採用
- **案 C**：このセッションだけ `export ANTHROPIC_API_KEY=...` する → 利用者が試したが効果なし（Claude Code が上書きする可能性あり、要再検証）。不採用

### 4.3 第 1 回ケース固有のプロンプト本文（議論論点 1 用、v3 確定版）

```
あなたは、ソフトウェアの仕様文書のレビュー結果を踏まえ、修正方針を決める最終判断者として依頼されました。

あなた（最終判断者）の役割は、3 役レビューで集まった所見と判定結果を踏まえ、判定役が「必ず直す」または「直した方がよい」と分類した所見ごとに、具体的な修正方針を決めることです。候補案の中から最適なものを選ぶか、新たな案を提示するか、情報不足のため深掘り要求を出すかを判断します。判定役の分類結果は尊重しますが、修正方針の最終決定はあなたが行います。

# 文脈

「ReviewCompass」は、ソフトウェアの仕様文書（intent ／ requirements ／ design ／ tasks ／ implementation の 5 段階）を段階的に書きながら、各段階で複数の視点でレビューを行う仕組みのシステムです（仕様駆動レビューと呼ぶ方式）。

レビューは「3 役レビュー」と呼ぶ方法で行います。同じ文書を 3 つの異なる立場の役（提案役・反論役・判定役）が独立に読み、意見を出し合います：

- **提案役**：文書を読み、問題点を網羅的に挙げる。あらかじめ決められた観点ごとに体系的に検査し、所見には重大度（CRITICAL／ERROR／WARN／INFO の 4 段階）と事実根拠（文書内の出典行）を必ず付ける。提案役は「漏れなく見つける」ことを優先する立場
- **反論役**：提案役のレビュー結果を読み、各所見を 1 件ずつ検証する。「同意（反証材料なし）」と「反証あり」の 2 択で応じ、反証ありの場合は具体的な根拠を示す。さらに、提案役が見落とした観点を独立に発見し、追加の所見として挙げる。反論役は「提案役と異なる視点で読み直す」ことを優先する立場で、提案役と異なるモデルを使う規律がある
- **判定役**：提案役と反論役の全所見（合計で 10〜30 件程度）を一括で読み、各所見について次の 2 つを決める。(1) 修正必要性の判定：「必ず直す（must-fix）」「直した方がよい（should-fix）」「直さなくてよい（leave-as-is）」の 3 段階に分類。(2) 波及種別の判定：「機能内対処（当該文書だけ直せばよい）」「波及（他機能の文書にも影響する）」「遡及（上流の文書を直す必要がある）」「延期（将来の段階で対処）」のいずれかに分類。判定役は「採否と影響範囲を決める」立場で、利用者と議論すべき重要な所見を明示する

現在、ある機能の「設計を実装作業に分解する段階（tasks 段）」で 3 役レビューが完了しました。判定役は所見 21 件を判定し、「必ず直す」10 件・「直した方がよい」7 件・「直さなくてよい」4 件と分類しました。

今回あなたに依頼するのは、その「必ず直す」または「直した方がよい」と判定された所見のひとつについて、具体的な修正方針を最終判断することです。

# あなたへの依頼

上記の所見について、対処方針を次の 4 つから 1 つ選び、理由を添えて回答してください：

- **採用：案 1** — 案 1 を採用する
- **採用：案 2** — 案 2 を採用する
- **別案を提示** — 案 1 とも案 2 とも違う案を新たに提案する
- **深掘り要求** — 判断するには情報不足、追加の確認や調査が必要

# 対話の進め方

不明な点があれば、判断を出す前に質問してください。最大 5 ターンの対話を許容します。十分な情報が揃ったと思ったら判定を出してください。

# 出力形式

decision: "採用：案 1" | "採用：案 2" | "別案を提示" | "深掘り要求"
rationale: |
  （判断の理由を平易な日本語で 3〜10 行）

  （decision が「別案を提示」のときのみ、新規案の内容）
confidence: 0.0〜1.0
turns_used: <実際に使ったターン数>
uncertainty_factors:
  - （判断に残る不確実性の要因 1）
  - （判断に残る不確実性の要因 2）
assumed_context:
  - （判断にあたって暗黙に置いた前提 1）
  - （判断にあたって暗黙に置いた前提 2）
case_scores:
  case_1: 0〜10（案 1 への評価点）
  case_2: 0〜10（案 2 への評価点）
comment_to_human:
  - （人本人への伝言、自由記述）
```
