# 実験データセット管理方針（長期開発における正本）

最終更新：2026-05-29（セッション40 初版）

本文書は ReviewCompass の実験データ（7 モデル比較／三役レビュー集計／事後発見指標など）の**長期管理の正本**である。論文の再現性と開発の継続性を両立させ、「現状分析したい」要求にも応える仕組みを定める。

## 0. 位置づけと優先順位

- 本文書が**実験データ管理の唯一の正本**である。
- 関連文書（実験ノート、評価インデックス、TODO 等）の記述が本文書と食い違うときは**本文書を優先**する。
- 本文書自体の改廃は §9 の手順に従う（軽微な追記＝CHANGELOG／リンク追加は本文書内で完結、主原則の変更は利用者明示承認）。

## 1. 背景と目的

3 つの要求が相反する：

- **論文用データの再現性**：固定された参照点が必要
- **開発の継続性**：仕様・実装・実験条件が時間とともに変わる
- **「現状分析したい」要求**：その時点の数値も取りたい

本文書はこの 3 つを両立させる仕組み（**タグで論文版を固定／開発は main で継続／集計は版非依存スクリプトで現状分析可能**）を規定する。

## 2. 基本原則

1. **論文版は git タグで固定**：タグは動かない印。main の進行から独立して参照できる。
2. **集計は版非依存**：集計スクリプト群（`tools/experiments/_aggregate_*_temp.py`）は `results/` を読むだけ。タグでも HEAD でも将来の HEAD でも同じスクリプトで動く。
3. **条件メタ情報を継続記録**：版の差を後から判別できるよう、データ収集条件の変化（判定支援資料の有無／バイアス緩和策／モデル世代／マルチターンの導入等）を §7 CHANGELOG に逐次記録する。
4. **新規データは条件を明示**：データを増やす際、その時点の条件を必ず注記。
5. **外部評価はタグ付け後に依頼**：動いている main で評価されると論文版とずれるため、タグを打ってから外部 AI 評価インデックスを渡す。
6. **論文版と現状分析を混ぜない**：現状分析の出力は「現状分析（YYYY-MM-DD 時点、HEAD コミット `xxx`）」として明記し、論文版の集計値と混在させない。

## 3. 版の命名規約

**タグ**：`paper-data-vN-YYYY-MM-DD` 形式（例：`paper-data-v1-2026-05-29`）
- N：版番号（v1, v2, …）
- YYYY-MM-DD：版を切った日付
- 用途別タグが必要な場合は派生サフィックスを許可（例：`paper-data-v1-2026-05-29-tasks` で「tasks 段までの版」と明示）

**ブランチ**：原則 main で開発を進める。並行開発が必要なら `paper-v1-fixes` 等の固定ブランチを切れる（論文受理後の差し替え修正等）。

## 4. 版を切るタイミング

次のいずれかで版を切る：

- 論文投稿時／改訂時／受理時
- フェーズ完了時（tasks 完了、implementation 完了等）
- 大きな発見や方法の変更が確定したとき（例：枠組み伝染バイアス対策の導入確定）
- 外部評価依頼の直前

## 5. 版を切る手順（チェックリスト）

1. 作業ツリーがクリーンであることを確認（`git status`）。push 済みであることを確認（`git log --oneline origin/main..HEAD`）。
2. main の最新状態でタグを打つ：
   ```
   git tag -a paper-data-vN-YYYY-MM-DD -m "<版の主旨と範囲を1〜2文>"
   ```
3. タグを push：
   ```
   git push origin paper-data-vN-YYYY-MM-DD
   ```
4. **実験ノート（`docs/experiments/n-model-comparison.md`）の「データ版の管理」節**に1行追加（版番号・タグ名・コミット・日付・データ範囲・主な発見）。
5. **評価インデックス（`docs/experiments/evaluation-index-for-external-review.md`）の §0** に「**現在の論文版**：タグ `paper-data-vN-YYYY-MM-DD`」を明記。
6. 本文書 §7 CHANGELOG に「版 vN を切った」旨と当時の条件を追加。
7. （必要なら）TODO_NEXT_SESSION.md の現在位置欄に版を反映。

## 6. 現状分析の手順

「いま時点の数値が欲しい」場合：

1. `git checkout main`（または最新ブランチ）に切り替え。
2. 集計スクリプトを再実行：
   - `_aggregate_agreement_all_temp.py`（モデル別人本人一致率）
   - `_aggregate_metrics_all_temp.py`（別案率／質問返し率／分岐度）
   - `_aggregate_calib_scores_temp.py`（confidence 較正／case_scores 分離度）
   - `_aggregate_triad_review_temp.py`（三役レビュー集計）
3. 出力結果は「**現状分析（YYYY-MM-DD 時点、HEAD `<コミット>`）**」として明記。論文版の集計値と混ぜない。
4. 論文版との差分が知りたいときは：
   - 生データ差分：`git diff <論文版タグ> HEAD -- tools/experiments/results/`
   - 集計値の差：論文版タグの集計表（実験ノート）と現状分析の集計表を並べて比較
5. 比較時は §7 CHANGELOG を参照し、版間の条件差を踏まえて解釈する（単純な数値比較は条件が同じ範囲に限る）。

## 9. 本文書の改廃

- **軽微な追記**（CHANGELOG への条件記録の追加、関連文書リンクの追加、用語の補足）は本文書内で完結。コミット時に「最終更新」日付を更新。
- **主原則・命名規約・手順の変更**は利用者明示承認が必要（governance 文書のため）。計画書 §5.12 改訂と連動して扱う場合は §5.12 改訂のスコープに含めて処理する。
- 本文書の場所（パス）は固定する：`docs/experiments/dataset-management.md`。移動・改名する場合は他文書のリンクをすべて更新する。
