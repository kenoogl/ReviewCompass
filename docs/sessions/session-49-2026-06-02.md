## 2. 気づき・特筆点

### 2.1 起草者のプロンプト不備による偽陽性（最大の発見）

初回 triad-review でレビュー対象に T-001 成果物の `docs/operations/RUNTIME.md` を含め忘れ、敵対役・判定役が「RUNTIME.md が存在しない」と must-fix を誤検出した（A-001 初回）。ファイル実在を確認のうえプロンプト生成を修正し、RUNTIME.md を含めた完全な対象で全3役を再実行した。レビューの網羅性は起草者のプロンプト設計に強く依存する。

### 2.2 起動手順の拡張解釈（恒常手順を直近の文字面より下に置いた）

冒頭「TODO を読む」指示に対し、§1 起動手順（利用者の指示を待たず全実行）を実行せず「読み上げのみで止めるか」と質問した。恒常的手順を直近の限定的発言の文字面より下に置く取り違え。規律 standing-directives-are-hard-constraints の裏返しの失敗。

### 2.3 push の明示承認（「OK」を push 含むと拡張解釈し拒否された）

コミット承認「OK」を push の承認も含むと拡張解釈して push を試み、auto mode classifier に soft block で拒否された。push（外部送信・不可逆）はコミット承認とは別の明示承認（「プッシュ」等）が必要。2.2 と同根の拡張解釈。

### 2.4 429 = insufficient_quota（レート制限でなく残高不足）

post-write-verification の3体目（OpenAI GPT-5.5）が HTTP 429 を3回返した。管理ページにレート制限はなく、レスポンス本文の error.type が `insufficient_quota`（残高・請求の問題）だった。残高追加で解消。429 はレート制限と残高超過の2系統があり、本文で切り分ける。

### 2.5 実行時ルールの書き込み検証の穴 → 対象範囲 案A（次セッション）

今回直接編集した実行時ルール（post-write-verification-spec.yaml＝.reviewcompass/specs/ 配下／config/api-settings.yaml＝config/ 配下）が、いずれも post-write-verification の対象範囲外で検証を受けていなかった。利用者指摘により対象範囲を拡大（案A）する方針を決定、規律改訂は次セッション（TODO §3 持ち越し）。
