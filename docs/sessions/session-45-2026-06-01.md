## 2. 気づき・特筆点

### 2.1 書き込み後検証の逸脱と回復（重要）

方針メモの独立検証で、Agent ツール（Claude モデルのみ呼べる）で haiku と sonnet を使い、**同一系統（Anthropic）内で検証してしまう逸脱**を犯した。規律「同一系統内代替は既定で認めない」に違反。利用者の指摘「逸脱行為ではないか」で発覚し、API スクリプト（`tools/api_providers/`）経由で OpenAI・Google 系統による再検証を実施して回復した。

- **教訓**：独立系統の検証は最初から API スクリプト経由で行う。Agent ツールは Claude ファミリー限定。
- **利用者の指摘**：「報告がないとお手上げ」。逸脱は事後報告があれば再実行で回復できるが、報告がなければ不正な検証が通ったまま進む。報告の徹底が要。

### 2.2 レビューが機能した実例（起草者と判定者の分離）

主役の唯一の重い指摘（P-001 ERROR：review_case.schema.json に x-deferred 注記がない＝規約違反）が、敵対役・判定役の論証で**覆った**。理由は「deferred（先送り拡張点＝必須リストに入れない項目）」と「責務分離（保持方法を runtime に委ねること）」の混同。step_records／findings は必須項目なので x-deferred は不要であり、起草実装は規約準拠だった。起草者（私）が判定に入らない仕組みが、誤った指摘を独立に打ち消した。

### 2.3 実装レビュー観点が実装前の不整合を検出

レビュー観点4（配置・命名規約）が、実装前から存在した FOUNDATION.md の配置記述の不整合（`schemas/`・`templates/` プレフィックスが design 段で `runtime/` に再編されたのに運用文書が追従していなかった）を検出。独立検証がさらに見落とし（ツール本体側の「リポジトリ直下」表記）も炙り出した。

### 2.4 検証スクリプトの偽陽性（テスト網羅性）

triad-review で、検査スクリプトが規約の一部を取りこぼす偽陽性が複数指摘された。must-fix の P-002（入れ子の required を検査しない）を対処。一方 A-001（deferred 注記の要否検査）は design.md が「検査機構は設計しない」と明記しているため触らず（無理に実装すると任意項目 override_reason の誤検出を招く）。
