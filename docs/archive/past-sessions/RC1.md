## 結論を先に：今やっているやり取りはレビューの実質を持っている

今セッションで進めてきた「判断根拠を整理 → 選択肢提示 → 利用者判断 → 計画書に記録」の繰り返しは、計画書 §5.9 の 3 役レビュー機構の構造と本質的に同じことをしています。違いは「形式的にレビュー記録として構造化されていない」だけ。

## 3 役のどれが動いているか

§5.9.1 で定義された 3 役と、今の実際の動きを対応付けます。

- **主役（一次検出）**：私（LLM）が担当
  - 例：「対象利用者の特定」で 3 つの選択肢を抽出し、影響度・整合性を分析した
  - 例：「§5.10.5 削除提案」で 4 シナリオを精査して内容上の根拠なしと結論した
- **敵対役（独立挑戦）**：明示的には立っていない
  - ただし、利用者が「精査」と返した場面では、私に敵対役の自己反論をさせる動きが起きた
  - §5.10.5 削除提案の精査では、反対根拠を意図的に探した（これは敵対役の動き）
- **判定役（採否決定）**：利用者が担当
  - 例：4 件すべての利用者判断、§5.19.6 の運用ルール「LLM は本一覧の判断を単独で確定しない」に従っている

通信順序も §5.9.1 の β（逐次）方式（主役 → 敵対役 → 判定役）に近い。

## 含意（解釈問題への影響）

仰った観点を取り込むと、先ほどの「解釈 α／β／う」の整理は次のように変わります。

- 抽出作業は **解釈 (う) 折衷案** に自然に収まる
  - 内容的には要件作成相当（解釈 β）：旧仕様を一般化して新仕様にする
  - 形式的には別途運用正本も作る（解釈 α）：`docs/operations/<機能>.md`
- そして **§5.9 規律の自己適用** がもう一段乗る：抽出作業の各節目をレビュー記録として構造化保存

## 1. 作業ディレクトリを ReviewCompass に移す件

### 仰る方針は §2.1 原則そのもの

§2.1（再構築の基本方針）に次があります。

> - 現リポジトリ＝素材として保全。変更は加えない（または最小限）
> - 新リポジトリ ReviewCompass＝再構築物として独立
> - 両者を混ぜない

実は今日のセッションで私たちはこの原則から外れ、`Rwiki-v2-code-mod` 側に計画書の更新コミットを 3 件積んでいます。仰る通り、ここで切り替えるのが筋。

### 切り替えで決める必要があること

新リポジトリで作業を始めるにあたって、計画書（`reviewcompass-reconstruction-plan-2026-05-21.md`）の扱いを決める必要があります。

- **案 (い)**：計画書を ReviewCompass に **移動**。今後の更新はすべて ReviewCompass 側。Rwiki-v2-code-mod 側は凍結（読むだけ）
- **案 (ろ)**：計画書を ReviewCompass に **複写（スナップショット）**。「移管時点の版」として記録し、今後の更新は ReviewCompass 側。Rwiki-v2-code-mod 側は歴史的記録として残す
- **案 (は)**：計画書は Rwiki-v2-code-mod 側に残し、ReviewCompass からは「外部参照」する。但しこの場合 §2.1「両者を混ぜない」原則から外れる

§9 関連で「計画書本体のどの部分を移管し、どの部分を旧リポジトリに残すか」が §5.19.5 に「移管時に判断」として記録されています。今がその移管タイミングという捉え方もできます。

### 素材の読み取りについて

ReviewCompass で作業しても、素材（dual-reviewer-rebuild/）は `Rwiki-v2-code-mod/dual-reviewer-rebuild/` に残ったまま、絶対パスで読みに行けます。書き込みは ReviewCompass 側のみ。読み書き境界が明確になります。

## 2. ログ自動記録の件

### 結論：自動記録に切り替える方が筋

仰るとおり、利用者が手動で逐一書き写すのは手間でもあり、抜けや改変リスクもあります。

### 自動記録の選択肢

- **(A) Claude Code のフック機能を使う**：`UserPromptSubmit` や `Stop`（Claude が返答を終えたとき）フックでセッションログをファイルに自動追記する。シェルスクリプトを書く必要があるが、一度設定すれば自動
- **(B) Claude Code のセッション記録（transcript）を流用**：`~/.claude/projects/.../*.jsonl` に session 全体が自動保存されている。これを後処理スクリプトで読みやすい Markdown に変換する
- **(C) 既存のフックなど活用なし、ターミナルのスクロールバックを後で保存**：最も簡便だが、長セッションだと取りこぼしの恐れ

§5.9 規律との関係を考えると：

- **生記録（raw evidence）**：(A) または (B) で自動取得。`evidence_register.json` に相当する保管庫
- **構造化レビュー記録**：節目ごとに私（または利用者）が手動で要約構造化（severity・judgment・evidence_type 付き）。生記録から派生する形

つまり生記録は自動、構造化記録は半自動（必要に応じて Claude が生記録から構造化を生成）が現実的です。

### 既存ファイル `docs/ReviewCompass再設計.md` の扱い

これまで利用者が手動で書き写してきた `docs/ReviewCompass再設計.md` は、自動記録への切り替え時点でどう扱うかを決める必要があります。

- **案 (a)**：このファイルは凍結し、以降の自動記録は別ファイル（例：`reviews/sessions/session-NN.md`）に蓄積
- **案 (b)**：このファイルを引き続き追記対象とし、自動記録もここに集約
- **案 (c)**：このファイルの過去内容を ReviewCompass 側に移管し、新規記録は ReviewCompass の構造化フォルダに保存

## 整理：今 1 度に決めるべき判断

仰る方向で進めると、次の判断が必要になります。

1. 計画書の扱い（案 い／ろ／は）
2. ログ自動記録の経路（案 A／B／C）
3. 既存 `docs/ReviewCompass再設計.md` の扱い（案 a／b／c）

これらを 1 つずつ確認していきますか、それともセットでまとめて判断したいですか。あるいは「まず計画書だけ決めて、ログの話は別途」のような分け方でも構いません。

なお、これは大きな方針転換に当たるため、決定前にもう一段確認しておきたい点があります。仰る「ワークフローに沿って進める」を厳密に取ると、**この決定自体も 3 役レビュー＋判定の対象**になります（§5.9 規律の自己適用）。私（主役）が論点と選択肢を整理し、私（敵対役）が反論を試み、利用者（判定役）が確定する、という流れ。今そのまま進めてしまってよいか、それとも先ほどの §5.19.6 運用ルールに「以降のセッション 18 内決定も §5.9 規律で構造化レビュー記録を残す」を先に確定するか、ここも判断が要ります。
