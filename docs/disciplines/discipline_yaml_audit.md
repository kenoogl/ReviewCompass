レビューの仕組みの根幹を決める設定・動作仕様 yaml（ワークフロー5段の成果物ではなく、5段の外側で直接編集されるもの）への書き込み後、書いた本人とは別系統の検証者による監査を必ず実施する。本規律は md（規律・仕様・計画）を対象とする書き込み後検証（[[post-write-verification]]）とは別立てで、yaml の性質に応じた観点で点検する。動作の詳細（対象範囲・観点・収束基準・初回検証・段階づけ）は動作仕様ファイル `.reviewcompass/specs/workflow-management/yaml-audit-spec.yaml` を正本とし、本規律はそれを参照する。

**Why**：レビューの仕組みの根幹を決めるのは yaml（設定・動作仕様）だが、これまで yaml には監査の規定がなかった。md には書き込み後検証（[[post-write-verification]]）があるが、yaml は性質が異なる（人が読む文書ではなく機械が読む構造化データ）ため監査の観点も異なり、md 用の規律をそのまま当てられない。無監査の yaml が誤っていると、レビュー機構そのものが静かに壊れる（例：規律本体が「yaml にある」と参照する節が、実物の yaml に存在しない不整合が、md 用検証では捕まらない）。監査されていないこと自体が重大な悪影響を持つため、yaml 専用の監査を新設する。2026-06-02 セッション50 で利用者決定。

**適用範囲（対象＝場所パターン方式）：**

5段の外側で直接編集される設定・動作仕様 yaml を対象とする。明示列挙ではなく場所と拡張子で機械判定する（対象が増えても自動で拾うため）。

- 対象に含める：
  - `config/**/*.yaml`（API などの設定）
  - `runtime/config/**/*.yaml`（reviewcompass.yaml などの設定）
  - `stages/*.yaml`（直下のみ。ワークフロー段の定義）
  - `.reviewcompass/specs/**/*.yaml`（各機能の動作仕様）
- 対象から除く：
  - `runtime/`（`config/` を除く）配下＝ runtime 実装の成果物（実装フェーズの3役レビューで検証）
  - `tools/`・`docs/` 配下＝実験データ・記録
  - `stages/completed/`・`stages/in-progress/` 配下＝実行時の状態ファイル
  - `*.md`・`spec.json` ＝ 5段ワークフローの成果物（別経路で検証）
- 補助ルール：
  - specs 配下の yaml は動作仕様とみなす（5段成果物は md と spec.json。yaml を成果物にしない前提）
  - 設定ファイルは `config/` か `runtime/config/` に置く規約とし、置き場所のばらつきによる漏れを防ぐ
  - 判定に迷う新規ファイル・新規ディレクトリは対象側に倒す

**監査の観点（11 項目、2 系統）：**

A 系統（機械で静的に検査できる。必須①〜⑥／推奨⑦⑧）：

1. 構文妥当性：yaml として壊れず読めるか
2. 構造・必須項目：期待する節・キーがそろっているか
3. 値の妥当性：列挙値が決められた語彙に収まるか
4. 参照の実在：他ファイルを指す参照先が実在するか
5. 規律・計画書との節一致：規律・計画書が「yaml にある」と書く節と実物が一致するか
6. 単一正本・重複整合：同じ動作を複数ファイルに書いた場合に値が一致するか
7. （推奨）下流コード・テストへの波及：節名・キー・値を変えたとき、読むコード（`tools/`）やテスト（`tests/`）が追従しているか
8. （推奨）検証テストの存在：その yaml の構造・値を検査する自動テストがあるか

B 系統（意味・原則の判断。独立検証で点検）：

9. 合意反映：書かれた値が利用者と合意した内容どおりか
10. 安全側既定・規律との非矛盾：既定値が安全側に倒れ、確定済み規律（検証者の独立性・起草者バイアス排除等）に反していないか
11. デプロイ／開発文書の分離：製品に同梱される yaml に開発経緯を書きすぎていないか、動作定義が開発文書側に残っていないか

**監査の方法：**

- A 系統（機械検査）：機械的なスクリプトで検査する（段階2以降）。スクリプトが未整備の段階1では、起草者と異なる別系統モデルが A 系統の観点も点検する。
- B 系統（独立検証）：md 用の書き込み後検証（[[post-write-verification]]）の検証者を流用し、別系統モデルが意味・原則を点検する。
- 検証者の独立性：起草者が自分の書いた yaml を自己点検しない（起草者バイアスの回避）。検証は必ず起草者と別系統が行う。
- フォールバック：必要な独立系統が揃わない場合は人へエスカレートする（[[post-write-verification]] のフォールバック節および各 yaml の応答可用性規定と同原則）。

**⑦⑧の確実な必須化（3点セット）：**

⑦⑧は当初「推奨」だが、人の記憶に頼らず確実に必須化するため次を据える：

1. 状態への条件連動（中核）：⑦⑧は機械検査スクリプトが利用可能になった時点で自動的に「必須」へ切り替わる（人の作業を介さない）
2. 忘れの自動検出（保険）：スクリプトが存在するのに⑦⑧が推奨のままという矛盾を、監査が警告（不合格）として検出する
3. リマインダの多重化：「段階2で⑦⑧を必須化」を `TODO_NEXT_SESSION.md`／本規律の段階づけの節／動作仕様 yaml の3か所に記録する

**初回検証（新規対象の組み入れ時。遡及とは別概念）：**

新たに監査対象へ組み入れる yaml は、組み入れ時に全件を初回検証する。yaml はこれまで監査体制の外にあり一度も検証されていないため、これは「過去への遡及」ではなく「これまで無監査だったものを初めて監査する初回適用」である。md 用書き込み後検証の「遡及しない原則」は既存の監査体制に対する過去分の扱いであり、本規律の初回検証とは別物。ここで「遡及しない原則」を持ち出すのは筋違いであり、監査されていないこと自体の悪影響を優先して全件を初回検証する（2026-06-02 セッション50 利用者裁定）。

**段階づけ（[[post-write-verification]] と同構造）：**

- 段階1（現状）：書き込み完了を認識したら、別系統モデルが観点①〜⑪を点検。止めるかは起草者の自律判断
- 段階2（将来）：A 系統①〜⑥の機械検査スクリプト（`tools/`）＋自動テスト（`tests/`）を整備。**⑦⑧は専用スクリプトの整備と同時に自動必須化**（スクリプトが利用可能になった時点で required へ自動昇格）。書き込み完了時に書き込み後フックで検証を起動し差し戻す
- 段階3（将来）：commit／push 段で Git／IDE フックにより未検査の書き込みを機械的に差し戻す（段階2の書き込み完了時フックに対し、本段は commit／push 時の最終関門）

**収束基準：**

検出時の処理（逐語的指摘は弾く／本質的指摘は人へ上げる／収束の定義）は、md 用書き込み後検証と同じ考え方を採り、動作仕様ファイル `yaml-audit-spec.yaml` の収束基準に従う。

**How to apply：**

- 対象 yaml の書き込み完了を起点に、別系統の検証者で観点①〜⑪を点検する（段階別に手段が異なる、上記段階づけ）
- 検証者には合意内容（決定の箇条書きのみ）と書き込み結果のみ渡す（[[post-write-verification]] の情報最小化と同原則）
- 検出は収束基準で分類処理（逐語は弾く、本質は人へ）
- 新規対象の組み入れ時は全件を初回検証する

**初回検証と発効後の運用：**

- 発効時点（本規律新設時）に監査対象へ組み入れる既存 yaml は、発効時に全件を初回検証する（合意案3：対象4件すべて）。
- 発効後に新たな yaml を監査対象へ追加する場合も、追加時に全件を初回検証する。
- 発効後の監査対象 yaml に誤りが見つかった場合は、発見の都度、再オープン手続き（`docs/operations/REOPEN_PROCEDURE.md`）で修正する。
