ユーザーに判断を仰ぐ際の選択肢提示は、ラベル明示と階層性を守る。

**Why:** 大規模 spec レビューでは判断ポイントが複数発生し、私が独断で選択肢を絞り込んだり、細部選択肢を一度に多数並列提示するとユーザーが判断不能になる。

- 2026-04-26 Spec 4 review で選択肢のある判断 (drafts SSoT 判定 / 修正先選択 / 用語統一方針 / 範囲拡張) を私が独断で 1 案に絞って提示した結果、ユーザーから「私のレビューが入っていない」と指摘されて approve 取り消し
- 同 Spec 4 review で C-1a / C-1b / C-2 / C-3 / C-4 / X-a-1 / X-a-2 / Y / Z を順次・断片的に提示した結果、ユーザーから「複雑すぎて判断が難しい」「判断を仰ぐときは大局的な観点から」とフィードバック

**How to apply:**

## 選択肢ラベル必須運用

修正案を提示する際は以下のラベルを必ず明示する。承認のみで適用可なものと判断ポイントのあるものを区別する。

- **【文言確定済】**: 唯一解で文言調整不要、承認のみで適用可
- **【選択肢あり】**: (a) / (b) / (c) など複数選択肢を提示、ユーザーが選ぶ
- **【SSoT 判定要】**: drafts / steering / 他 spec で記述が複数あり、どれを正解とするか不明、ユーザーが SSoT を確定する
- **【範囲拡張】**: 既存記述に明示されていない新規要素を追加する場合、ユーザー確認

ユーザーが「全件 (a) で承認」等の一括判断を返すことは可能だが、私が独断で選択肢を 1 つに絞り込んで提示してはならない (ただし `feedback_deepdive_autoadopt.md` の自動採択対象に該当する場合は深掘り検討の結果を報告した上で自動採択可)。

## 判断ポイントの典型 (見落とさない)

- drafts 自体の内部矛盾の SSoT 判定 (例: drafts §7.2 内で表記が逆転しているケース)
- 修正先選択 (本 spec を直すか他 spec を直すか drafts を直すか)
- 用語統一方針 (CLI 名 vs 操作名 vs 併記、命名ポリシーは v2 全体に波及)
- 他 spec 改修を今回やるか TODO 化するか (dependency 順序と spec phase の判断)
- design phase に流す範囲 (要件レベルで確定するか design 持ち越しか)
- 既存記述漏れを補う場合の範囲拡張 (暗黙の追加を避ける)

## 階層性ルール (大局 → 細部)

判断を仰ぐときは **大局的な観点から先に問い、必要に応じて細部に降りる**。一度に多数の細部選択肢を並列提示しない。

- **第 1 階層 (大局)**: 「本セッションのスコープを X に保つか Y に拡大するか」「修正方針を A 軸 vs B 軸のどちらに統一するか」のような **方針レベル** の質問を先に出す
- **第 2 階層 (細部)**: 大局判断が決まってから、その方針内の具体的選択肢 ((a) / (b) / (c) 等) を提示
- **同時並列回避**: 大局判断と細部判断を **同じターンで並列に出さない**。並列に出すとユーザーが「大局を選んだ後の細部影響」を見通せず判断できない

## 提示の具体ルール

- **選択肢の数**: 1 ターンで提示する選択肢は **3 個以内** が目安。4+ 並ぶ場合は大局判断を分離する設計が間違っている可能性を疑う
- **メリデメ説明の長さ**: 各選択肢のメリット・デメリットは **箇条書き 2-3 点** に絞る。長文比較表は判断負荷を上げるだけ
- **私の推奨**: 提示時に必ず私の推奨を明示し、ユーザーが「推奨で進める」と一言で答えられるようにする

## 物理 layout / path / structure 選択時の例外 (concrete artifact 提示)

ファイル配置 / directory 構造 / path 命名等の物理 layout 選択時は、上記「メリデメ箇条書き 2-3 点」ルールを次の 3 点組みで置き換えると判断負荷が下がる (2026-04-30 12th セッション dual-reviewer-foundation design phase で user 認証):

- **concrete tree 提示**: ASCII directory tree で各候補の actual layout を示す (abstract path 文字列より concrete)
- **比較表**: 5 観点以内の比較軸 (例: navigation / 既存規約整合 / Phase 移行性 / 命名統一 / 純度) で各候補を ✅ / △ / ❌ で評価
- **判断軸要約**: 「X 重視 → 候補 1 / Y 重視 → 候補 2」の形で大局的判断軸を最後に明示

理由: path / layout は physical artifact = abstract 説明より concrete tree の方が判断 input として明瞭、比較表は判断軸が固定 (location 選択の直接関連軸) のため負荷上昇しない。general feature 選択 (algorithm / pattern / strategy) では従来ルール (箇条書き 2-3 点) を維持。

## 典型 anti-pattern

- 「(a)/(b)/(c)/(d)/(e) と新案 (F)/(G) も追加で検討してください、各メリデメは下記...」 → ユーザーは細部で迷子になる
- 「C-1a / C-1b / C-2 / C-3 / C-4 / C-5 を順次判断してください」 → 大局合意なしに細部を積み上げる構造
- 「dominated 案も含めて全選択肢を公平に提示」 → `feedback_dominant_dominated_options.md` 違反、判断負荷を上げるだけ
