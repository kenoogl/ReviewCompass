ReviewCompass の不可逆操作（spec.json approve／commit／push／フェーズ移行／規律変更等）は利用者明示承認必須。

**承認の判定基準：**

- **承認とみなす**：明示的肯定発言（「承認します」「OK」「採用」「進めて」「はい」「ア」等）、または AskUserQuestion での明示選択
- **承認とみなさない**：議論継続・停止・方法論指示・沈黙・命名指摘

**確定記述には出典必須：**

「確定」と書く各項目に、利用者の明示承認発言の出典（日付・正確な引用）を併記する。出典なしの確定記述は禁止。

**機械検査は承認の代替ではない（2026-05-25 セッション 24 段階 3 導入後）：**

段階 3 フック（補助層 C）の exit 0（OK）判定は「機械的な依存・整合・安全性が通過した」を意味するだけで、「利用者承認が取れている」とは別問題。承認の明示発言と出典の引用は依然として LLM の責務であり、フック通過を承認の代替として扱わない。

**Why:** 旧 3 規律（approval-required／no-implicit-approval／explicit-approval-citation-required）を統合（2026-05-25 セッション 24）。承認の運用が複数の規律に分散していたため、文脈圧力下で見落とすリスクがあった。集約することで承認の規律を一体で読み返せる形にした。過去の失態：approval 未取得で design 提案、議論継続を承認と誤認、出典なし「確定」記述の常態化。

**How to apply:**

- 不可逆操作の直前：承認の明示発言を待つ。曖昧な発言は確認を取る
- 確定事項表（TODO §4、議論メモ、計画書改訂履歴等）の各項目：必ず出典発言を引用
- 一見承認に見える発言（「いいかも」「進めても」等）でも明示性に疑念があれば確認
- 機械検査の補強として段階 2 スクリプト（[[workflow-precheck-invocation]]）の commit／push チェックが部分対応、未対応部分は依然として LLM が守る
- 詳細・過去事例：`memory/archive/2026-05-25-consolidation/` 配下の旧 3 ファイルを参照
