## 1. 問題

LLM（メインセッション）が、ワークフローの処理手続きを正しく把握しないまま「次のステップ」「反映の段取り」を提案する失敗が**何度も繰り返されている**。

セッション 39 の具体例（self-improvement tasks の triad-review 後の反映の段取り）：

- **1 回目の誤り**：依存順を無視して「tasks を先に直し、後で設計・要件」と提案した。運営ガイド §3.3 を**確認せずに**提案した（§0.1 違反）。
- **2 回目の誤り**：利用者に「ワークフローに沿うか」と問われて初めて §3.3 を読み、今度は「上流に戻って大がかりに再オープン、tasks は後回し」と**過剰修正**した。「機能内の所見は当該段で今処理する」基本ルールと workflow-management セッション 38 の先例（遡及 A-001／A-003 を同セッションで処理）を結びつけられなかった。

過去にも同型の失敗（ワークフロー把握不足のまま着手）が繰り返されており、運営ガイド自体がセッション 19 の同種失態を防ぐために作られた経緯がある。

## 2. 真の原因

ワークフロー手続きが「散文（読んで解釈する文章）」として複数文書（運営ガイド §2.3／§3.3、計画書 §5.x、規律ファイル）に**散在**している。そのため毎回「読む → 解釈する → 今のケースに当てはめる」が必要で、解釈を誤れば失敗する。**LLM の記憶・判断に依存する限り、ケースが増えるほど必ず破綻する**。

ケースごとに規律・フックを足す対処は、**組み合わせ爆発でメモリ肥大**を招く対症療法であり、根本ではない。

## 3. 合意した根本解（ブートストラップ完了＝workflow-management 実装フェーズで対応）

プロジェクトの設計思想「**真実源はファイル、判定は機械、LLM の記憶に頼らない**」（spec.json が状態の正本、`tools/check-workflow-action.py` が不可逆操作の機械ゲート）を、**ワークフローのナビゲーション**（次に何をするか／所見をどう振り分けるか）にも適用する。

1. **手続きを一つの問い合わせ可能な形に集約する**
   運営ガイド §2.3（段の進め方）＋§3.3（機能内／波及／遡及の振り分けと処理段）を**一枚の決定表**（状態×出来事 → 次の処理）に統合する。あるいは既存の `tools/check-workflow-action.py` に問い合わせ機能を追加する：
   - `next`：今の workflow_state から次に許される処理を出力
   - `classify`：所見種別（機能内／波及／遡及）→ 処理段を出力
   **新規ツールを増やすのではなく、既にある 1 つを拡張する。**

2. **規律を N 個 → 1 つのメタ規律に畳む**
   ケース別の規律を足すのをやめ、「**ワークフローの進行・反映の段取りを提案する前に、必ずナビゲータに問い合わせ、その出力を応答に引用する**」という 1 つのメタ規律にする。これは全ケースを覆い、ケースが増えても規律は増えない。§0.1 はこれを目指しているが「ガイドを読め」という散文依存なので、「**この 1 つのナビゲータに問え**」に作り変えるのが要点。

3. **本質は workflow-management 機能そのもの**
   いま仕様を書いている workflow-management は、段集合の静的定義・完了述語・再オープンの trigger_map（再実施対象の決定表）・依存マップで、**手続きを機械が管理・強制する**機能である。**根本解＝workflow-management の実装**。現状の繰り返す失敗は「その機能がまだ仕様段階で、手続きが散文のままだから」という**ブートストラップ期（自分を作る途中で、自分の道具がまだ無い）の症状**である。

### 7.1 複数機能の同時再オープンの挙動（未定義）

発見：2026-05-29 セッション 40、A-018（foundation 語彙正本件数の自己矛盾）の対処検討中。利用者の問い「補足修正を evaluation/design.md に置くと、foundation の再オープンと evaluation の再オープンが同時にオープンの状態になるか、その場合の挙動はどうなるか」。

現状：REOPEN_PROCEDURE.md（暫定版）は**単一機能の再オープンを前提**にしており、複数機能が同時にオープン状態になる場合の挙動を明示的に定義していない。今回は補足修正を evaluation/tasks.md 側に置く案（あ）を選び、オープンを foundation 1 件に限定して同時オープンを回避した。

未定義の論点（workflow-management 設計で詰める）：

- **進行中状態ファイルの命名衝突**：`stages/in-progress/reopen-procedure-<日付>.yaml` は日付ベース。同日に複数機能を再オープンするとファイル名が衝突する。機能名サフィックス等で一意化する規則が要る
- **並行管理の単位**：複数の独立した再オープンを同時に走らせるか、直列化するか。spec.json は機能ごとに別ファイルなので技術的競合はないが、停止点（各機能の approval・コミット）が機能数だけ増える
- **recheck フラグの相互作用**：foundation は全機能の土台。foundation の design 再オープンが他機能へ波及した場合、各機能の `recheck.upstream_change_pending` をどう連動させるか（現状は機能ごとに手動）
- **共通土台（foundation）再オープン時の波及確認**：foundation の語彙正本そのものの値を変える再オープンでは、全下流機能の参照整合を alignment 段で確認する必要がある。この一斉確認をどう機械化するか

当面（ブートストラップ期）：単一機能のオープンに限定できるよう修正の置き場所を選ぶ（今回の案あ）。やむなく複数同時になる場合は進行中状態ファイルを機能名で一意化し、機能ごとに第1〜4過程を独立に回す。
