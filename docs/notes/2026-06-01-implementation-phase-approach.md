## 3. 実装フェーズのゴール（4点）

1. **タスク文書との整合**：成果物（コード・YAML・JSON 等）が tasks.md の完了条件どおりに実装されている
2. **テスト通過**：設計されたすべてのテストが pass している
3. **要件に従った動作と意図適合**：実装が要件文書を経て意図文書まで追跡でき、意図どおりの機能を発揮している
4. **配置可能性**：成果物が対象アプリの `.reviewcompass/` 配下に展開でき、他機能から参照可能な形式・場所に置かれている（計画書 §5.23.7）

---

## 4. 実装レビュー観点（暫定5点）

計画書 §5.9.2 は「実装適合：5観点」と件数のみ規定し、内容はスタブ。
本セッションでの dogfooding（自己適用検証）により以下を暫定確定。

| 番号 | 観点 | 主に問うこと |
|---|---|---|
| 1 | タスク文書との整合 | 成果物の構造・内容がタスク文書の完了条件と一致しているか |
| 2 | 要件への端々の追跡 | 実装成果物が要件追跡表（tasks.md §要件追跡）を経て要件本文まで辿れるか |
| 3 | テスト網羅性と信頼性 | テストがタスク文書のテスト要件を網羅し、偽陽性（実際は壊れているのに通る）がないか |
| 4 | 配置・命名規約の遵守 | 成果物の場所・ファイル名が design.md §共有資産配置および tasks.md の成果物欄と一致するか |
| 5 | 機能横断波及の早期検出 | 実装の決定が他機能（runtime 等）の仕様や配置規約に影響しないか |

**利用者明示承認の出典**：「暫定で推薦案を採用」（2026-06-01 セッション45）

将来改訂条件：foundation の triad-review で観点が不足または過剰と判明した場合、
再オープン手続きを経ずに暫定改訂可（まだ確定版ではないため）。

## 5. 実装不能の扱い

「実装不能」（実装を進めた結果、設計の前提が成り立たないことが判明した状態）は
独立カテゴリを設けず、既存の2枠に吸収する：

- **設計上の欠陥**が原因の場合 → 遡及（上流への手戻り）として再オープン手続きを起動
- **実装手段の問題**（設計の前提は正しいが選んだ方法では実現できない）の場合 → triad-review（3役レビュー）の所見として対処方針を議論

**利用者明示承認の出典**：「既存枠に入れる」（2026-06-01 セッション45）

## 7. 実装の進め方（テスト駆動開発）

各タスクの実装順序：
1. テストを先に書く（失敗することを確認）
2. テストをコミット（テストのみ、実装コードは書かない）
3. テストを通す実装を書く
4. すべてのテストが pass することを確認してコミット

依存順（tasks.md より）：
T-001 → T-002・T-003（並行可）→ T-004・T-005（T-003 完了後）→ T-006・T-007（T-001 完了後）→ T-008・T-009・T-010
