# コンテキスト上限到達時のセッションログ異常系

## 発生状況

Claude のコンテキスト上限に達してセッションが引き継がれた際、セッションログファイルが想定外のタイミングで更新され、push フックに遮断される状態が生じた。

## 何が起きたか

通常のセッションではフックは次の順に起動する。

```
SessionStart → 作業 → SessionEnd
（前セッションログを staged に追加）  （今セッションログを書き込み）
```

コンテキスト上限に達すると、引き継ぎの境目で SessionEnd → SessionStart が割り込む。

```
SessionStart → 作業 → [上限到達] → SessionEnd → SessionStart → 作業 → SessionEnd
                                    ↑ この時点でセッションログが書き込まれる
```

今回の具体的な流れ：

1. セッション冒頭の SessionStart フックが前セッションのログを staged に追加した
2. 私たちがそれをコミットした
3. コンテキスト上限到達による中途 SessionEnd が走り、同じログファイルに追記された
4. 結果：「コミット済みだが内容が変わった」（`M` 状態）になった
5. git push を実行 → push フックの precondition「作業ツリーが clean であること」を満たさず遮断された

## 根本原因

`session-record-capture.sh`（SessionEnd フック）は、コンテキスト上限による引き継ぎと、通常のセッション終了を区別しない。どちらも同じようにログファイルを書き込む。

## 今回の対処

`git status` でログファイルの未コミット変更を確認し、別途コミットしてから push した。

## 設計上の問題点

- push フックの precondition「作業ツリーが clean」は、コンテキスト引き継ぎという外部要因を考慮していない
- セッションログは自動生成ファイルであり、通常の作業変更とは性質が異なるが、現状は同列に扱われている

## 改善候補

| 案 | 概要 |
|---|---|
| A | SessionEnd フックで引き継ぎか通常終了かを判別し、引き継ぎ時はログの書き込みを抑制または後回しにする |
| B | push フックの precondition を「自動生成セッションログファイルは例外とする」に緩める |
| C | push 前に自動生成ファイルを自動でコミットするステップを挟む |

案 A が根本的だが、引き継ぎの検出方法が未確定。案 B・C は対症療法だが実装は簡単。
