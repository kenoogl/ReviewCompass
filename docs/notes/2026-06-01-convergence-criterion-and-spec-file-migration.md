## 2. 収束基準の論点と確定内容

出発点はセッション 42 の試行（`docs/notes/2026-05-31-item2-proxy-and-postwrite-verification-trial.md`）で観察した「1 系統が逐語厳格で収束しにくい」という挙動にある。実質的な誤りは 1 回目の検証で捕捉済みなのに、Anthropic Sonnet だけが文体・好みレベルの指摘を出し続け、旧規律「1 件でも検出で一律阻止」を適用すると収束しない恐れがあった。

**確定した収束基準：**

- 検出を「逐語的指摘（弾く）」と「本質的指摘（人へ上げる）」の 2 種類に分類する。
- 切る軸は「深刻度（高・中・低）」ではなく「修正の性質」。機械的に直せる字句レベルか、判断を要する本質かで切り分ける。
- 系統数（多数決）では切らない。少数の系統だけが本質を突くことがある。実験ノート §3.4.2 の topic-102 が典型例——7 モデル中 6 が起草者の枠組みに同調し、1 モデルだけが計算式の不条理を見抜いた。多数決なら不条理な式が通っていた。
- 逐語的指摘：機械的に直せれば直し、直せなければ記録に留める。再検証ループは回さない。修正後は起草者が機械的反映を 1 回だけ確認する（`post_fix_recheck`）。
- 本質的指摘：起草者が独断で修正せず人へ上げる。逐語か本質か迷えば本質側に倒す（安全側原則）。
- 収束＝未解決の本質的指摘がゼロ。逐語的指摘だけが残る状況は実質収束。

**旧仕組みの廃止：**

旧 `retry_policy.max_retries`（人なしの自律的な差し戻しループ、既定 5 回）を廃止した。設定名 `max_retries` は計画書 §5.9.7 の API 通信リトライ専用に戻した（同名異義の解消）。

## 3. デプロイ問題と動作仕様ファイル方式

当初、収束基準を計画書 §5.8 に直接記述した。しかし計画書は開発文書でデプロイされない。規律ファイル（デプロイ対象）が計画書を動作仕様として参照すると、デプロイ時に参照先が消えるデッドリンクになる、という問題を利用者が指摘した。

**解決策：動作仕様のみを記述したコンパクトな専用ファイルを作成し、デプロイ対象に配置する。**

- 作成ファイル：`.reviewcompass/specs/workflow-management/post-write-verification-spec.yaml`
- 形式は YAML を選択。将来フックスクリプト等の機械が読んで動作を制御する用途を想定しており、Markdown より機械パースが確実なため。判断規範のニュアンスは説明フィールド（`description:` の複数行文字列）に散文で記述した。
- 動作仕様ファイルは「現在の動作の定義」のみを持つ。経緯（確定日・移行措置）は持たない——これは計画書 §5.8 に置く役割分担。
- 計画書 §5.8 と規律本体（`discipline_post_write_verification.md`）は、動作仕様ファイルを参照する形に改訂した。

**派生課題：**

他の規律も同じデプロイ問題を抱える。該当箇所：post_write_verification の §5.13（通知機構）・§5.6（再オープン手続き）への参照、workflow_state_truth_source の §5.24（spec.json スキーマ）への参照。「動作仕様ファイル方式」の適用を検討する積み残し課題として TODO §3.6 に切り出した。なお、説明・経緯としての参照（デプロイ時に参照先が消えても動作に影響しない）は対象外。

## 4. ドッグフーディング（自己適用）2 回の観察

新収束基準を確立する過程で、確立したばかりの基準を即座に自己適用した。

**第 1 回（計画書 §5.8 への書き込み後、実機 3 系統）：**

- Anthropic Sonnet：ALL_CLEAR
- OpenAI GPT-5.5：DETECTED（5 件）
- Google Gemini-3.1-pro-preview：DETECTED（5 件）

検出の大半は、検証プロンプトに既存事実・判断理由を渡さなかったための逐語的指摘だった（日付・キーパス・段数など）。これらは新収束基準で弾いた。本質的指摘 1 件（「初期値は上記列挙とする」が合意の射程内か曖昧）は人が裁定し「初期値として確定」で収束した。

**第 2 回（計画書 §5.8 を参照化＋規律本体改訂の書き込み後、実機 3 系統）：**

- OpenAI GPT-5.5：ALL_CLEAR
- Google Gemini-3.1-pro-preview：ALL_CLEAR
- Anthropic Sonnet：DETECTED（1 件）——「再検証ループは回さず、修正時は起草者が機械的反映を 1 回確認」が経緯の混入との指摘

本質的指摘 1 件（規律本体に経緯を書くかの判断）を人が裁定し「現在の動作記述として残す」で収束した。

**観察のまとめ：**

新収束基準により、旧基準（1 件でも検出で一律阻止・最大 5 回ループ）なら差し戻しになっていた事案が、逐語を弾き本質 1 件ずつを人裁定で収束した。セッション 42 で観察した「収束しにくい」問題への対処として機能した。

**検証プロンプト設計の教訓：**

情報最小化を厳しくしすぎる（決定事項のみ渡し既存事実・判断理由を渡さない）と逐語的指摘が増える。日付・キーパス・段数などの既存事実を参考欄に入れるべき。実験ノート §3.4.2 と同型の教訓（情報不足による質問返しの防止と同じ構造）。

## 5. メタ観察：ツール呼び出しの不安定

Opus 4.8 の 1M コンテキスト版＋長大コンテキストで、ツール呼び出しの malformed（開始タグの破損）が頻発した。Sonnet サブエージェントへの委譲で回避した。本試行記録の作成自体も Sonnet が担当している。新規コンテキストでは安定する見込み。
