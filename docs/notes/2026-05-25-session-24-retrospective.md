## 2. 規律違反の記録

### 違反 1：「次セッションで」終結誘導

**発生**：セッション 24 中盤、補助層 C 共存モデルの議論メモ・計画書 §5.8・TODO §3 D／§4 を反映した直後（コミット `04a2eef` 後）。

**内容**：私が応答末尾で「(イ) を推奨：本セッションはここで区切り、次セッションで段階 2 仕様策定」と勝手に session 終結を誘導する選択肢を提示。利用者の判断を仰がず LLM 側から終結方向に誘導していた。

**利用者の指摘**：「セッション継続します。勝手に終わらせない」（2026-05-25 セッション 24）

**対処**：
- memory `feedback_no_unilateral_session_ending.md` を即時新設（規律として保全）
- MEMORY.md active 必読層に追加（19 件 → 20 件）
- 規律内容：「session の継続／終結は利用者判断、LLM が『本セッションはここで完了』『次セッションで』等の終結誘導をしない」
- その後、規律統廃合（セッション 24 後半）で旧 [[no-unilateral-approach-change]] と統合し、新規律 [[no-unilateral-action]]（勝手に走らない、approach／session 含む）として一体化

**再発防止**：統合規律 [[no-unilateral-action]] が「session 継続終結」も明示対象とすることで、approach 変更と同列の「勝手に走らない」原理として扱う形に整理。次セッション以降の運用で監視する。

### 違反 2：（観察対象）応答末尾の誘導が再発しないか

統合後の [[no-unilateral-action]] の運用継続。本セッション後半では「次の選択肢」の提示時に「本セッションを区切る」を含めない運用に徹した（観察結果として）。

## 4. 学んだこと

### 規律統廃合の効果

- 機械化可能な規律（事実確認・workflow_state 読み・依存チェック等）と意味判断必須の規律（approach 判断・session 進行・応答スタイル等）の責務分離が明確化
- 統合により Why／How to apply の重複が圧縮され、実測 48.7% のトークン削減
- 個別ニュアンスの欠落リスクは archive 保全で緩和（旧 14 ファイルは `memory/archive/2026-05-25-consolidation/` に保全）

### 共存モデルの妥当性

- 段階 1（LLM 規律）と段階 3（フック）は責務が異なる：段階 1 は応答テキストのみの判断（フックが効かない領域）、段階 3 はツール呼び出し時の強制発動
- 段階 2（外部スクリプト）が両者の機械的判定基盤として機能
- 当初の「3 段階を時間軸の段階的導入計画」から「最終形態として共存する役割分担」へ位置付け変更が功を奏した

### TDD の徹底

- 4 ラウンド（段階 2 spec-set／commit／push テスト＋実装の各往復、段階 3 テスト＋実装の往復）すべて TDD 規律「テスト先行・実装ロジックは書かない」を遵守
- テストヘルパーの setup バグ（pending ファイル未追跡、tools/ 未追跡）も assertion 不変のまま是正

### 段階 2 スクリプトの実運用

- 1 セッション内で 14 シナリオの実測を実施、想定どおりの判定（OK／WARN／DEVIATION）
- 初の WARN 観察：計画書編集の commit で要注意変更を検出（コミット `c57b837` 直前）
- 規律 [[workflow-precheck-invocation]] が機能、利用者の再承認運用も正しく回った
