### 共存モデルの核（採用済み方針）

3 段階を **役割分担して共存** させる：

| 段階 | 単一責任 | 主体 |
|---|---|---|
| 段階 1：意図宣言と結果解釈 | これから何をするかを応答内で明示、段階 2 を呼び、出力を解釈して次の行動を決める | LLM |
| 段階 2：機械的判定 | spec.json／git／規律／持ち越し所見を読み、引数の処理が現状で適切かを返す | 外部スクリプト |
| 段階 3：強制発動 | ツール呼び出しの前に段階 2 を自動で走らせ、逸脱なら遮断 | Claude Code フック機構 |

段階間のインターフェース：段階 1 と段階 3 の両方が段階 2 を呼ぶ。段階 1 は能動的・宣言的、段階 3 は受動的・自動的。

### 段階 1 を恒久層として位置付ける（重要差分）

旧メモ（セッション 23 末作成）では段階 1 を「段階 2 完成までの過渡期措置」と位置付けていたが、共存モデルでは **段階 1 を恒久層** とする。理由：

- 段階 3（フック機構）はツール呼び出し時点で発動するため、**応答テキストのみの判断**（例：「完了」「承認済」の無根拠断定）には効かない
- この領域は段階 1 が恒久的に担う必要がある
- 段階 1 は段階 2 完成後に薄く縮約するが廃止しない

### 既存規律との関係（部分採用）

(3) 18 件 → 12 件の規律統廃合方針は **本承認には含まれない**。個別規律ごとの判断を要する別議論として、段階 2 が動いてから本格検討（本メモ §「規律統廃合の可能性」の方向性は参考扱い）。

### コスト見積もり（議論で提示）

- 1 セッションあたり LLM 出力トークン費用増：$0.15〜0.75 程度（Opus 4.7 換算）
- 実行時間：段階 3 を絞り込めば +10〜80 秒程度
- 応答冗長化：段階 1 宣言で 5〜10% 増
- 利用者の手戻り対応負担：減（逸脱予防効果）
- 実装の一時コスト：1〜2 日相当
- 数値はすべて見積もりであり、段階 2 のスクリプト実装後に小規模運用して実測することが必要

> 「ワークフローの手順の逸脱防止案を考えてみた。何かの処理を始めたとして、その処理がトリガーとなって、ワークフローの手順をチェックさせることはできるか。そうすれば、これからやろうとしている処理が手順に適ったものかどうかがわかる。もし逸脱していれば自ずから修正がはいることが期待できる。」

本質：処理が呼ばれた時点で、その処理が現在のワークフロー手順に適合するかを **事前検査** し、適合しなければ遮断または警告する仕組み。LLM の自己申告性に頼らず、構造的に手順遵守を強制する設計。

## 実装案 3 段階

### 段階 1：軽量（処理前自己検査チェックリストの規律化）

CLAUDE.md または規律ファイルに「処理前自己検査チェックリスト」を新設し、私（LLM）が処理を始める前に応答内で必ず明示宣言する：

1. 何の処理をしようとしているか（具体的に）
2. 現在の workflow_state（実 Read で確認した結果）
3. その処理が現在の workflow_state で適切か
4. 不可逆操作なら、直前に利用者の明示承認発言があるか

**強制性**：弱（LLM の自己申告に依存）
**実装コスト**：低
**LLM 自己申告依存度**：高

### 段階 2：中程度（外部スクリプトによる検査）

`tools/check-workflow-action.py` のような外部スクリプトを整備：

- 引数：これからの処理の種類（`edit-requirements`／`approve-requirements`／`commit`／`push` 等）と対象機能
- スクリプトが spec.json／規律ファイル／pending-cross-feature-findings.md を読んで判定
- 適切なら終了コード 0、逸脱なら非 0 と逸脱理由を出力

LLM が処理前にこのスクリプトを Bash で呼ぶ。逸脱検出時は処理を止めて利用者に報告。

**強制性**：中（LLM が呼ぶ前提）
**実装コスト**：中
**LLM 自己申告依存度**：中

### 段階 3：重量（Claude Code のフック機構を使う）

Claude Code の PreToolUse hook（Edit／Write／Bash 等のツール呼び出し前に外部スクリプトを自動実行）を活用：

- `.claude/hooks/pre-tool-use.sh` を設定
- LLM がツールを呼ぶたびに自動でチェックスクリプトが走る
- hook が止めれば処理は遮断される（fail-closed）

**強制性**：強（強制発動、LLM が呼ばない選択ができない）
**実装コスト**：高
**LLM 自己申告依存度**：低

## 規律統廃合の可能性

段階 2 を導入すれば、現在の active 規律（18 件程度）のうち次は **部分代替** できる：

| 規律 | 段階 2 での代替方法 |
|---|---|
| [事実はログ・git で確認] | スクリプトが spec.json／git log を読んで現状を返す |
| [完了検証プロトコル] の機械照合部分 | スクリプトが grep／spec.json チェックを自動実行 |
| [承認なしで進めない] | スクリプトが直前の利用者発言と spec.json 状態を照合 |
| [確定事項表には承認出典必須] | スクリプトが「出典が書かれているか」を形式チェック |
| [workflow_state が唯一の真実源] の読む側 | スクリプトが workflow_state を常に読み、要約を返す |
| [複数 file 操作前 graph 調査 v2.1] の grep 部分 | スクリプトが grep を自動実行し結果を整形 |
| [集計局面の自己検査チェックリスト] | スクリプトが 5 項目を機械チェック |

### 置換が難しい規律

これらは LLM の意味判断・応答スタイル・文脈判断に依るため、機械化困難：

- [意図適合が受け入れ基準]
- [恒常指示は硬い制約]
- [勝手に走らない]
- [事実と解釈を明示的に分ける]
- [簡潔・もれなく報告]
- [ジャーゴンを避け平易な日本語で書く]（部分チェックは可能）
- [AskUserQuestion を多用しない]
- [セッション開始時の引継ぎ規律]
- [論点の再オープン手順]
- [reactive 書き直し]
- [黙示承認の不採用]

### 統廃合の具体的提案（次セッションでの検討事項）

段階 2／3 を導入した場合に検討できる統廃合の方向性：

1. **[事実はログ・git で確認] ＋ [workflow_state が唯一の真実源] の読む側 ＋ [完了検証プロトコル] の機械照合**
   → 統合して「**処理前後に外部スクリプトを必ず呼ぶ**」という 1 規律にまとめる

2. **[承認なしで進めない] ＋ [黙示承認の不採用] ＋ [確定事項表には承認出典必須]**
   → 段階 3 フックに統合し、規律としては「**不可逆操作は段階 3 フックを必ず通す**」に簡素化

3. **[複数 file 操作前 graph 調査] ＋ [集計局面の自己検査チェックリスト]**
   → 統合して「**集約・横断操作の前に検査スクリプトを必ず呼ぶ**」に簡素化

これにより、現在 18 件程度の規律を **10〜12 件** に圧縮し、各規律の長さも 1／2 〜 1／3 に短縮できる見込み。

## 規律削減の効果（計画書 §5.8 第 5 層と整合）

計画書 §5.8 第 5 層「処理表面積の抑制方針」と整合する効果：

- 規律の総語数が減れば、LLM が文脈圧力下で抱える規則の量が減る
- 結果として各規律の遵守度が上がる可能性
- 段階 2／3 の機械強制と相補的に働く

## 残る限界（正直な共有）

どの段階でも完全には解決しない問題：

1. **LLM の意図の解釈**：処理に渡す引数（「これから何をしようとしているか」）の正確性は依然として LLM の規律遵守に依存
2. **静的検査の限界**：機械的に判定できない曖昧な状況が残る（例：「approval 前の最終調整」と「無断書き換え」の境界）
3. **処理表面積の増加リスク**：規律を追加するほど LLM が抱える規則の量が増え、文脈圧力下で他の規律遵守が下がる可能性
4. **逸脱を「修正させる」のは別問題**：検査が遮断しても、その後の修正は LLM の判断

## 派生論点（次セッションでの検討候補）

- **規律の階層化**：機械化可能な規律（スクリプトに移譲）と意味判断必須の規律（LLM が守る）の明確な分類軸を導入
- **規律の自動生成**：外部スクリプトの仕様変更に伴い、関連規律も自動更新される仕組み（CLAUDE.md の自動生成）
- **規律遵守率の計測**：段階 2 のスクリプト呼び出し回数を計測すれば、LLM の規律遵守度を機械的に測れる（self-improvement の効果測定 7 指標と接続）
