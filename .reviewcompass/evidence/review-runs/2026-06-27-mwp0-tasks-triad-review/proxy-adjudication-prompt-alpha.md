# proxy_model 判断依頼：クラスタ α（テスト変更禁止 vs 既存テスト更新の矛盾）

## 依頼の位置付け

workflow-management 機能の tasks フェーズ triad-review（MWP-0 追記分）における must-fix クラスタの採否判断を依頼する。あなたの役割は「どの修正案を採用するか」を決定することである。実装は担当しない。

## 問題の説明

tasks.md には現在 2 つの矛盾した記述が存在する。

**T-015（既存タスク）のテスト要件節**：
> テストは `tests/tools/test_phase1_schema_definitions.py` に作成済みで commit 済み（失敗状態）。実装でテストを通過させる。**テストの変更は禁止。**

**T-020（MWP-0 で新設したタスク）のテスト要件**：
> `next_action_response.schema.json` の `kind` enum が7値に限定されていること（**既存テストの更新**）

T-015 のテスト変更禁止は、当時のテストが kind を「14値」として検証していたため設けられた制約である。しかし MWP-0 によって kind が14値から7値へ変わったため、テストの更新は設計上必要である。T-020 がそのテスト更新を担うが、T-015 の「禁止」制約がそのまま残っているため、実装担当者が矛盾に直面する。

## 元 review raw への参照

- claude-sonnet-4-6.round-1.txt（WARN）：「T-015 のテスト変更禁止」と「T-020 の既存テスト更新」の関係が文書上明確でないと指摘
- gpt-5.5.round-1.txt（WARN）：「MWP-0 により kind enum が変わるためテスト更新自体は妥当。ただし T-020 による例外・上書きであることを明記すると矛盾を避けられる」と指摘
- gemini-3.1-pro-preview.round-1.txt（ERROR）：「テスト変更禁止と既存テスト更新の矛盾が実装フェーズで混乱やコンフリクトを招き、想定外のテスト破壊につながるリスクがある」と指摘（3モデル共通の指摘）

## 候補案

**案 A（T-015 側に例外を明記）**

T-015 のテスト要件節を次のように修正する：
> テストの変更は禁止。**ただし kind enum 値の変更（MWP-0 受入 12 反映）に伴うテスト更新は T-020 が担う。**

- 利点：T-015 を読んだ実装担当者が例外をその場で把握できる。禁止の意図（kind 以外のテストは変えない）も維持される。
- 弱点：T-015 が T-020 に依存する参照を持つことになり、T-020 が将来削除・改名された場合に参照が腐る。

**案 B（T-020 側に制約の有効範囲を明記）**

T-020 のテスト要件の当該行を次のように修正する：
> `next_action_response.schema.json` の `kind` enum が7値に限定されていること（T-015 のテスト変更禁止は kind enum 値に限り T-020 で上書き更新する）

- 利点：T-020 内で制約関係が閉じており、T-015 を変更しなくて済む。
- 弱点：T-015 の禁止制約を読んだ実装担当者が「T-020 を見なければ例外を知れない」状態になる。T-015 の禁止は引き続き誤読リスクを持つ。

**案 C（両方に記述）**

案 A と案 B を組み合わせ、T-015 側に例外注記、T-020 側に制約範囲の明記を両方行う。

- 利点：どちらを先に読んでも矛盾を把握できる。最も冗長性が高い。
- 弱点：2 箇所の同期が必要になり、将来の保守コストが増える。

## 私（メインセッション LLM）の推薦案

**案 A**を推薦する。

T-015 は「テストの変更は禁止」という強い制約を掲げており、その制約文書を読んだ実装担当者が例外の存在を知るためには T-015 側への記載が不可欠である。案 B では T-015 の誤読リスクが残る。案 C は冗長だが、参照腐敗リスクの面で案 A より弱い。T-020 は今後も tasks.md に残り続けるタスクであるため、T-015 から T-020 への参照腐敗リスクは低い。
