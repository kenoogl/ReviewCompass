# tasks triad-review adversarial（gpt-5.5）round-1

## A-001 [ERROR] T-013 の完了条件が design §6 の implementation 委譲事項を取り込んでいない

**場所**: T-013 完了条件

**問題**: design.md §6「implementation へ委譲する未確定事項」は 4 項目を列挙している：
(a) `--verify-pending` ファイル更新時の安全性保証（並行書き込み防止・失敗時の不完全書き込み防止）
(b) `bundle_exception_id` 採番規則の実装細部
(c) 逐語不一致時の差分表示形式
(d) `decision-source-lint` の既存関数との統合方法

T-013 の完了条件はこれらをゲート化していない。特に(a)は安全性に関わり、(c)はユーザ体験に影響する。tasks 段の完了条件はこれらを「実装で確定すること」として明示すべきである。

**影響**: implementation 段でこれらが実装されても、tasks の完了条件に含まれていないため、完了判定の「全タスク完了」条件でこれらが要求事項として検証されない。

**分類**: ERROR

## A-002 [ERROR] テスト要件(2)の束ね例外チェックで「3 条件すべて満たす場合のみ通過」が実装要件として曖昧

**場所**: T-013 テスト要件(2)「束ね例外 3 条件チェック（承認レコード有＋covered_decision_ids 含む＋multiplicity=single で通過）」

**問題**: 3 条件の**部分的な満足**（例：承認レコードあり＋covered_decision_ids 含むが multiplicity=bundled のまま）でも fail-closed になることのテストが明示されていない。design §2 の「3 条件をすべて満たす場合のみ通過」は逆裏「1 つでも欠けたら fail-closed」のテストが必要だが、テスト要件は正常系のみ記述している。

**影響**: 部分的な例外条件で誤って通過するバグを見逃す可能性がある。

**分類**: ERROR

## A-003 [WARN] T-013 の責務①に設定ファイル「生成」とあるが、tasks 段でファイルを「生成する」か「定義する」かの区別が不明

**場所**: T-013 責務①「`stages/decision-source-lint-config.yaml` の生成（内容なし語リスト初期値 11 件）」

**問題**: 「生成」は implementation が行う行為であり、tasks（タスク定義）での記述としては「初期値 11 件の内容なし語リストを持つ設定ファイルを配置する」とすべき。「生成」という動詞では、タスクがコード生成なのか、ファイルを手動作成するのか、実装コードがランタイムで生成するのかが不明。

**影響**: 成果物の定義として「`stages/decision-source-lint-config.yaml`（初期内容なし語リスト）」とあり整合はしているが、責務記述の精度が低い。

**分類**: WARN

## A-004 [WARN] テスト要件にスキーマの「必須フィールド欠落時の挙動」のみ言及され、不正型の挙動が未規定

**場所**: T-013 テスト要件(1)「必須フィールドと category 3 値 enum の検査（欠落・不正値で DEVIATION）」

**問題**: category の不正値（enumに含まれない文字列）はカバーしているが、型不正（category: [list] など非文字列型）・decision_id が空文字列・source.excerpt が空文字列の場合の挙動が明示されていない。design §1 の型制約（category 3 値 enum のみ）から期待されるが、テスト要件としての明示がない。

**影響**: 空文字列・非文字列型の入力に対する実装が実装者の判断に委ねられる。

**分類**: WARN

## A-005 [WARN] 完了条件の「commit 直前ゲートへの統合済み」が integration test での確認であることを明示していない

**場所**: T-013 完了条件「commit 直前ゲートへの統合済み（`pending=WARN`・`unverifiable=DEVIATION`）」

**問題**: 「統合済み」という完了条件は、単体テストでの mock 確認なのか、実際の `check-workflow-action.py commit` コマンドを使った統合テストでの確認なのかが曖昧。テスト要件(10)で「`cmd_commit` が `decision-source-lint --all` を呼び出し...」とあるが、unit test レベルなのか end-to-end レベルなのかが不明。

**影響**: implementation 段で unit test のみで「統合済み」と判定し、実際の commit フローでの動作確認が漏れる可能性がある。

**分類**: WARN

## A-006 [INFO] テスト要件の番号体系が 11 項目あるが、design §1〜§5 の対応を追跡しにくい

**場所**: T-013 テスト要件(1)〜(11)

**問題**: テスト要件が線形番号だが、design §1（スキーマ）・§2（束ね例外）・§3（照合）・§4（内容なし語リスト）・§5（サブコマンド）・§6（委譲事項）のどれに対応するかが即座に分からない。ただし tasks 文書の他のタスク（T-012 等）も同様の線形番号を採用しており、スタイルの一貫性がある。

**影響**: 実装者がテスト要件と設計節の対応を手動で辿る必要がある。許容範囲。

**分類**: INFO
