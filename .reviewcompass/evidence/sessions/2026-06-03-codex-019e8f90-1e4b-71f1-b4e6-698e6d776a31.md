プロキシモードで判断理由・候補案・採用案を決める過程を詳しく説明

## アシスタント（2026-06-03T22:17:27.511Z）

プロキシモードでやることは、簡単に言うと **「人が判断するはずだった重要所見について、別モデルに“判断者”を任せる。ただし、判断過程をあとから追える形で残す」** です。

流れはこうです。

1. **レビュー結果を材料化する**

まず、各モデルから返ってきた raw レビューを読みます。  
ここでは parsed YAML だけに頼らず、raw 本文も対象にします。理由は、Claude などが構造化 YAML として壊れた返答をしても、レビュー内容自体は raw に残っている場合があるためです。

この段階で作る材料は次です。

- どのモデルが何を指摘したか
- severity は何か
- 指摘対象ファイル・箇所はどこか
- 何が問題だと言っているか
- 修正を求めているのか、注意喚起なのか
- 同じ問題を複数モデルが言っているか

ここではまだ実装しません。

2. **三段階トリアージする**

各所見を、正本語彙で次の三つに振り分けます。

- `must-fix`：このまま進めると仕様違反、データ不整合、後続処理の失敗、再レビュー必至になるもの
- `should-fix`：今直すのが望ましいが、ただちに破綻するほどではないもの
- `leave-as-is`：修正しない。誤検知、仕様上許容、または現段階の対象外

この分類は「モデルが ERROR と言ったから必ず must-fix」ではありません。  
ERROR でも前提誤認なら `leave-as-is` になり得ますし、WARN でも後続に効くなら `must-fix` になり得ます。

3. **重要件だけ判断モードに入る**

重要件とは、基本的に次です。

- `must-fix`
- severity が `ERROR` / `CRITICAL`
- 複数モデルが同根で指摘しているもの
- 上流仕様、データ契約、workflow 規律に影響するもの
- 実装後に手戻りが大きくなるもの

通常モードでは、ここで人に平易説明して承認を取ります。  
プロキシモードでは、この部分を **proxy_model が代行**します。

4. **候補案を複数出す**

proxy_model に渡す前に、または proxy_model 自身に求める形で、各重要件について候補案を作ります。

例：

- 案1：指摘どおり実装修正する
- 案2：仕様側の意図を確認し、テストだけ補強する
- 案3：今回は修正せず、`leave-as-is` として理由を記録する
- 案4：別タスク・後続フェーズへ延期する

重要なのは、候補案が一つだけだと判断にならない点です。  
「直すべきです。直します」では、人の判断代行になりません。比較が必要です。

5. **各案の利点と弱点を説明する**

proxy_model は、各候補について次を説明します。

- 何が良いか
- 何が危ないか
- 実装コストはどの程度か
- 既存仕様との整合はあるか
- 後続の review-wave / alignment / approval に影響するか
- テストで機械確認できるか
- 今回のスコープを超えないか

ここで重要なのは、**採用しない案も公平に扱うこと**です。  
最初から結論ありきで他案を弱く書くと、プロキシ判断としての独立性が落ちます。

6. **採用案を決める**

proxy_model は、最終的に一つを選びます。

記録すべき内容は次です。

```yaml
approved_by: proxy_model
proxy_model_id: gemini-3.1-pro-preview
approved_action: review_run_apply_fixes
review_run_id: ...
finding_id: ...
selected_option: option_1
final_label: must-fix
rationale: ...
rejected_options:
  - option_2: ...
  - option_3: ...
confidence: ...
```

ここでの `rationale` は「なぜその案を選んだか」です。  
`rejected_options` は「なぜ他案を選ばなかったか」です。

7. **人に見せるための平易説明も残す**

プロキシモードでも、判断内容は後から人が読める必要があります。  
したがって、重要件ごとに次のような説明を残すのがよいです。

- 問題：何がまずいのか
- 影響：放置すると何が起きるのか
- 候補：どんな直し方があるのか
- 推奨：どれを採るのか
- 理由：なぜそれが一番よいのか

これは「人に毎回承認を求める」ためではなく、**後から監査できるようにするため**です。

8. **機械ガードで通す**

最後に、triage が全部 `decided` になっているか、`must-fix` / `should-fix` の修正対象が承認済みかを機械判定します。

通常モードなら `approved_by: user`。  
プロキシモードなら本来は `approved_by: proxy_model` を許容し、さらに次を確認する必要があります。

- proxy_model が許可されたモードか
- main モデルと独立したモデルか
- raw レビューが参照されているか
- 三段階トリアージが完了しているか
- 重要件に判断理由・候補案・採用案があるか
- 採用案と `final_label` が一致しているか
- 未判断の `human_required` が残っていないか

現状の実装ではここがまだ `approved_by: user` 固定に寄っています。  
つまり、**手順としては定義できるが、完全な機械判定にはまだ proxy_model 用 approval record の実装が必要**です。

まとめると、プロキシモードは「LLM が勝手に直す」ではありません。  
正しくは、**別モデルが、レビュー raw を根拠に、候補案を比較し、採用案と理由を記録し、その記録を機械ガードに通してから実装するモード**です。

まず、**各モデルから戻ってきたレビューをまとめて三段階トリアージするのはメインセッション LLM** です。  
つまり、raw を読み、同根所見をまとめ、`must-fix` / `should-fix` / `leave-as-is` の下書きを作るところまでは、いま会話している LLM が担当します。

その後の流れは、判断主体ごとに分かれます。

1. **メインセッション LLM**
メインセッション LLM は、トリアージ結果を整理して、重要件ごとに判断材料を作ります。

やることは次です。

- 各所見を平易に説明する
- 何が問題かを整理する
- 候補案を複数出す
- 各案の利点・弱点を並べる
- 推薦案を出す
- 判断が必要な点を明示する
- proxy_model に渡す判断依頼プロンプトを作る
- proxy_model の回答を raw として保存する
- proxy_model の判断を構造化して記録する
- 最終的な実装修正リストに落とす

ここで重要なのは、メインセッション LLM は **整理者・実装者・記録者** であって、重要件の最終判断者ではない、という点です。

2. **proxy_model**
proxy_model は、重要件について **人間の判断を代行する判断者** です。

やることは次です。

- メインセッション LLM が作った判断材料を読む
- 元レビューの raw / 要約 / triage を確認する
- 候補案を比較する
- 採用案を 1 つ選ぶ
- 採用理由を書く
- 採用しなかった案の理由も書く
- 最終ラベルを確定する  
  例：`must-fix` / `should-fix` / `leave-as-is`

proxy_model は、実装はしません。  
あくまで「この所見はどう扱うべきか」を決める役です。

3. **機械ガード**
機械ガードは、proxy_model の判断が本当に揃っているかを確認します。

確認するものは次です。

- 未判断の所見が残っていないか
- `must-fix` / `should-fix` に判断記録があるか
- 判断者が許可された proxy_model か
- raw response が保存されているか
- 候補案・採用案・判断理由が記録されているか
- triage の最終ラベルと proxy_model の判断が一致しているか

ここを通らない限り、メインセッション LLM は実装に進めません。  
ただし現状の実装はまだ `approved_by: user` 固定が残っているので、proxy_model を正式に通すには改修が必要です。

4. **メインセッション LLM**
機械ガードを通過した後、メインセッション LLM が実装します。

やることは次です。

- 採用された `must-fix` / `should-fix` だけを修正対象にする
- TDD の手順でテストを書く
- 赤を確認する
- 実装する
- テストを通す
- 修正結果を triage / review-run に反映する
- 作業サマリを出す

ここでは、proxy_model が決めた方針を勝手に変えません。  
実装中に新しい判断問題が出た場合は、再度 proxy_model 判断または人間判断に戻します。

5. **人間**
このモードでは、人間は全所見の個別判断はしません。  
ただし、次は人間の職掌として残ります。

- proxy_model モードを使うこと自体の承認
- コミット
- プッシュ
- spec.json 更新
- フェーズ移行
- 規律変更
- 大方針変更
- 上流仕様に戻る reopen 判断

つまり、人間は「全部の細かい技術判断を読む人」ではなく、**不可逆操作と方針境界の承認者** になります。

まとめると、順番はこうです。

```text
レビュー取得
  ↓
メインセッション LLM が raw をまとめて三段階トリアージ
  ↓
メインセッション LLM が重要件ごとに判断材料と候補案を作る
  ↓
proxy_model が採用案・理由・最終ラベルを決める
  ↓
機械ガードが proxy 判断の充足を確認
  ↓
メインセッション LLM が TDD で実装
  ↓
人間がコミット等の不可逆操作を承認
```

要するに、**メイン LLM は整理と実装、proxy_model は判断、人間は不可逆操作と方針境界** です。

具体的に実装サブ担当LLMはどう扱う？サブエージェントかAPIでどこかに投げるか

## アシスタント（2026-06-03T22:17:27.512Z）

具体的には、**実装サブ担当 LLM は API 直投げではなく、サブエージェント扱いが本筋**です。

理由は、実装担当には次が必要だからです。

- repo を読める
- テストを実行できる
- ファイルを編集できる
- 差分を返せる
- 失敗時に赤/緑の状態を説明できる

API だけに投げると、基本的には「パッチ案を文章で返す」だけになり、実際の作業ディレクトリ・テスト・差分管理と切れます。なので API は **判断・レビュー・設計相談** には向きますが、実装サブ担当としては弱いです。

推奨はこの順です。

1. **サブエージェント / 別 Codex 作業単位**
   - 同じ repo か分離 worktree を持たせる
   - 実装単位 A だけを渡す
   - 対象ファイル、触ってよい範囲、該当 proxy decision、期待テストを明示する
   - 結果は差分とテスト結果で返す
   - commit / push / spec.json 更新は禁止

2. **別スレッド / 別 worktree**
   - 並列実装の衝突を避けやすい
   - ファイル単位で完全に分離できる場合に向く
   - 最終統合はメインセッション LLM が行う

3. **API 直投げ**
   - 原則は実装担当にしない
   - 使うなら「この関数の修正案を出す」「テストケース案を出す」程度
   - 返答は patch 候補として扱い、適用と検証はメインセッション LLM が行う

したがって正本上の扱いは、こう書くのがよいです。

> 実装サブ担当 LLM は、原則として repo 読み書きとテスト実行が可能なサブエージェントまたは分離作業スレッドとして扱う。API-only モデルは実装担当ではなく、実装案・テスト案・局所パッチ案の助言者に限定する。

実務上は、メインセッション LLM が「統括者」で、サブエージェントは「限定された実装作業者」です。  
proxy_model は判断者、API レビューモデルはレビュア、実装サブ担当は作業者、という分離です。

2. 各単位ごとに別スレッド・分離 worktree を作る
3. サブ担当には以下だけを渡す

   - 触ってよいファイル
   - 書くべきテスト
   - 実装禁止事項
   - 新しい判断問題が出たら停止する条件
4. サブ担当は commit / push / spec.json 更新をしない
5. メインセッション LLM が差分を取り込み、統合テストを実行する
6. 最終報告と不可逆操作の承認依頼はメインセッション LLM が行う
