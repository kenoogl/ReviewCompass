## アシスタント（2026-06-05T11:59:31.736Z）

参照しました。最後の議論では、SES2026は **三役レビューとプロキシモデル** が主眼で、AgenticDev2026ではそこから少しずらして、

> LLMエージェントを「自律開発者」ではなく、レビュー・証跡・状態遷移・人間承認を調整する **ワークフロー・コンシェルジュ** として設計する

という論旨にする案が出ていましたね。これはかなり良い軸だと思います。

AgenticDev 2026のCFP上も、multi-agent collaboration、orchestration、human-in-the-loop、既存ワークフローやツールチェーンへの統合、trustworthiness / transparency / evaluation が明示的に対象になっているので、この方向は会議趣旨に合っています。投稿締切は **2026年7月15日 AoE**、full 10ページ、short/demo/tool 5ページです。  
参照: [AgenticDev 2026 CFP](https://conf.researchr.org/home/ase-2026/agenticdev-2026)

僕なら、SES26との差分をこう整理します。

**SES26**
「LLMレビューをどう裁定するか」  
三役レビュー、プロキシモデル、判定統合、裁定負荷が中心。

**AgenticDev2026**
「LLMレビューを含む開発ワークフロー全体を、エージェントがどう運用可能にするか」  
コンシェルジュ、機械ゲートとLLM判断の分担、並列レビュー、合議、証跡管理、人間承認が中心。

つまり、SES26の成果を捨てるのではなく、AgenticDevではそれを **ワークフロー・コンシェルジュを構成する部品** として再配置するのが良さそうです。

論文の核は、次の一文にできそうです。

> Agentic software development requires not only coding agents, but also workflow concierge agents that coordinate reviews, evidence, machine-checkable gates, and human approvals, thereby balancing LLM flexibility with process accountability.

この主張なら、今回苦労した「機械判定とLLM利用の柔軟さのバランス」「並列実行」「複数ターンの合議」「複数ケース評価」が全部、単なる実装苦労話ではなく研究貢献に昇格します。

次に詰めるべきは、たぶん **投稿タイプ** です。  
5ページ demo/tool なら ReviewCompass を前面に出して「こういうシステムを作った」。10ページ full なら、複数ケースを使って「この設計原則がどの失敗を検出し、どこに限界があったか」まで言う。僕の感触では、締切までの時間を考えると **5ページ tool/experience寄り** が堅く、10ページ full は評価整理の工数次第です。

なので新規性は、**“workflow concierge” という名前そのもの**ではなく、もっと具体的な技術的差分に置くべきです。

僕なら、新規性候補を3つに絞ります。

**1. LLM判断と機械ゲートの分離設計**
単にLLMエージェントを並べるのではなく、

- LLMが判断するもの
- 機械が判定するもの
- 人間承認へ戻すもの

をワークフロー状態として明示的に分ける。  
特に、LLMの柔軟な所見をそのまま次工程に渡さず、manifest、review finding、approval record、state transition などの機械可読な証跡に落としてから進める点。

これは「agent orchestration」一般より具体的です。

**2. 例外・寄り道を正規化する状態モデル**
post-write-verification / BTW Track / dirty git tree の扱いは新規性になり得ます。

多くのLLM開発支援は、理想的な直線ワークフローを前提にしがちです。  
でも実際のagentic developmentでは、途中で文書を直す、証跡を補う、未承認変更が混ざる、レビュー中に仕様へ戻る、という例外が頻発する。

そこを、

> deviationとして単に失敗させるのではなく、対象変更・禁止変更・復帰先・完了条件を持つ補助トラックとして状態化する

というのはかなり良い主張です。

**3. 複数LLMレビューを「合議」ではなく「証跡付き状態遷移」に接続する点**
三役レビューやプロキシモデルだけなら、「LLMを複数呼びました」に見えます。  
でも、それらの出力を

- blocker / warning / passed
- unresolved / resolved
- evidence required / evidence satisfied
- human approval required
- next workflow action

のような機械判定可能な状態へ接続するなら、単なる ensemble ではなく workflow governance の仕組みになります。

ここを強く書くとよいです。

逆に、新規性として弱く見えやすいものは以下です。

- WebUIで進捗を見せる
- LLMレビューを複数モデルで行う
- 並列実行する
- 人間承認を挟む
- ログを残す

これらは単独だと既視感があります。  
ただし、上の3つの新規性を支える材料としては有効です。

一文で言うなら、新規性はこう置けます。

> Our novelty is not in adding another agent dashboard or multi-agent reviewer, but in modeling agentic development as machine-checkable workflow state transitions where LLM reasoning, hard gates, side-track recovery, and human approval are explicitly separated and coordinated.

日本語なら：

> 新規性は、LLMレビューや可視化UIそのものではなく、LLMの柔軟な推論、機械的ゲート、補助トラックによる復帰、人間承認を、機械判定可能なワークフロー状態遷移として統合した点に置く。

1. **A workflow-state model** for governed agentic development, separating LLM reasoning, machine-checkable gates, and human approvals.
2. **A side-track recovery mechanism** that normalizes interruptions such as post-write changes and dirty git states with explicit return points and completion criteria.

4. **A case-based evaluation** over ReviewCompass development episodes showing what failures were detected, what gates prevented, and where human judgment remained necessary.

## アシスタント（2026-06-05T12:11:40.412Z）

同意です。ここで必要なエビデンスは「ReviewCompassが便利だった」ではなく、**提案した状態モデルが、従来のLLMエージェント運用では見落としやすい失敗を、機械判定・証跡・復帰制御として扱えた**ことを示すものです。

新規性ごとに必要なエビデンスを分けるとよいです。

**1. LLM判断と機械ゲートの分離**
欲しいエビデンス：

- LLM判断だけでは曖昧だったが、機械ゲートで止められた事例
- 逆に、機械判定では扱えず、LLMレビューが意味解釈した事例
- 最終的に「LLM所見 → 機械可読な finding / manifest / approval / workflow state」に変換された記録

示したいこと：

> LLMと機械判定を混ぜるのではなく、役割分担すると運用可能性が上がる。

具体例候補：

- unresolved finding が残っているため完了不可
- evidence path が存在しないため gate failure
- approval record がないため次工程に進めない
- LLMは問題なしと言ったが、manifest sha256 不一致で止まった

**2. Side-track recovery / BTW Track**
ここが一番新規性を強く出せそうです。

欲しいエビデンス：

- 正規ワークフロー中に外側文書・証跡・メタデータを変更した事例
- その変更が post-write-verification によって検出された記録
- dirty git tree が target / non-target changes に分解された記録
- 検証完了後に元の workflow state へ戻れた記録
- もし補助トラックがなければ混線したであろう事例

示したいこと：

> 例外を単なる逸脱や失敗として扱うのではなく、復帰先と完了条件を持つ一時的な補助トラックとして正規化できる。

これはかなり刺さります。

**3. Multi-role / multi-agent review を状態遷移に接続**
欲しいエビデンス：

- 三役レビューまたは複数モデル判定の raw outputs
- それらが blocker / warning / passed などに正規化された記録
- 不一致が発生し、人間または追加ターンで裁定した記録
- 合議後に workflow state がどう変わったか
- 並列実行で時間短縮したログ

示したいこと：

> 複数LLMの出力を集めるだけでなく、証跡付きの workflow transition に接続している。

特に「合議した結果、何が次の機械的状態になったか」を見せたいです。

**4. Workflow Navigator / 可視化層**
ここは主張の補助なので、エビデンスも控えめでよいです。

欲しいエビデンス：

- state schema の例
- 現在位置、blocked reason、next action、return_to が一意に出る例
- 人間が判断すべき項目と機械的に済んだ項目を区別できる例
- 画面モックまたは実装スクリーンショット

示したいこと：

> 可視化は装飾ではなく、機械可読な workflow state を人間が運用するための control plane である。

**5. 複数ケース評価**
10ページfullなら、最後はケース表が重要です。

最低限、こういう表がほしいです。

```text
Case | Phase | Trigger | LLM role | Machine gate | Detected issue | Recovery action | Outcome
```

例：

```text
self-improvement | implementation review | multi-role review | semantic critique | unresolved finding gate | blocker remained unresolved | rerun review + update evidence | passed
workflow-management | post-write verification | doc change | explanation | sha256 manifest gate | stale manifest | regenerate manifest | returned to main workflow
conformance-evaluation | proxy decision | adjudication | role disagreement | decision schema validation | missing rationale | additional turn | human-approved
```

数としては、**3ケースでも深ければ十分**です。  
理想は5から7ケースですが、10ページでは表が膨らみすぎるので、本文では3ケース詳述、appendix/artifactで全件、がよさそうです。

一番大事なのは、各ケースで次を揃えることです。

- 何が起きたか
- LLMだけなら何が曖昧だったか
- 機械ゲートが何を確定したか
- 人間判断がどこに必要だったか
- workflow state がどう遷移したか
- 結果として何を防いだか、または何を回復したか

これが揃うと、新規性は「アイデア」ではなく「観測された設計上の必要性」として語れます。

僕ならまず、エビデンス収集の優先順位はこうします。

1. **post-write / BTW / dirty tree の復帰事例**
2. **LLM判断と機械ゲートが食い違った事例**
3. **三役・複数モデルレビューが状態遷移に接続された事例**
4. **複数ケースを横断した summary table**
5. **Navigator/WebUI は schema と1画面例に留める**

この順なら、論旨がぼけずに新規性を支えられます。

論文上は、評価を二段構えにするとよさそうです。

**Evaluation A: Dogfooding Study**
ReviewCompass自身の開発ログを使う。  
目的は、提案設計がどの問題から生まれ、既存dogfoodingで何を検出できたかを示す。

見るもの：

- LLMレビューと機械ゲートの役割分担
- multi-role review の不一致と裁定
- unresolved finding / evidence mismatch / approval missing
- 状態遷移への接続

**Evaluation B: Deployed Case Study**
デプロイ版を外部コード群に適用する。  
目的は、ReviewCompass固有のdogfoodingだけでなく、異なるドメインでも workflow concierge が機能するかを見る。

対象：

- phase field
- IoT
- thermal-fluid solver

見るもの：

- 初期仕様からレビュー完了までのワークフロー進捗
- machine gate が検出した停止条件
- LLMレビューが発見した意味的問題
- side-track / post-write / recovery の発生有無
- Navigatorで人間が現在位置と次アクションを把握できたか

新規性の示し方としては、本文でこう言えます。

> Dogfooding data demonstrates why explicit separation between LLM reasoning and machine-checkable workflow gates is necessary. Deployed case studies then test whether the resulting concierge workflow can guide review processes across heterogeneous technical codebases.

注意点は、デプロイ版評価で「コード品質が上がった」と言いすぎないことです。そこまで測ると大変です。  
主張すべきはむしろ、

- workflow stateが一貫して管理できた
- gate failure / blocked reason / next action が明示できた
- LLM判断が証跡付きfindingに変換された
- 例外作業から復帰できた
- 人間が承認すべき箇所を分離できた

です。

つまり評価指標はコード性能ではなく、**governed workflow quality** に置く。
