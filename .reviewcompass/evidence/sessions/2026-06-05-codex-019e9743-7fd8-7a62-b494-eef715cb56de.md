1. **ワークフロー地図**
   全体工程をノード/レーンで表示し、現在位置、完了済み、待機中、ブロック中、再オープン中を色と形で表す。

2. **現在の判断面**

3. **対話・ログペイン**
   LLMとの会話、実行ログ、判定履歴、手動承認履歴を同じ時系列で保持する。後から「なぜこの状態になったか」を追える。

まず決めるべき設計論点は、「UIが正本になる」のではなく、**既存のYAML/ログ/状態ファイルからUIが導出表示する**ことです。これなら既存の機械判定フローを壊さず、Web UIは観測・操作レイヤーとして育てられます。

段階案としては、

1. **静的ビューア**
   既存ファイルを読み、ワークフロー全体と現在位置を表示するだけ。

2. **リアルタイム更新**
   ファイル変更監視で、判定結果やログ追加を即時反映する。

3. **操作UI**
   承認、再実行、次ステップ生成、LLM呼び出しをUIから実行する。

4. **独立アプリ化**
   LLM対話、ログ保存、ワークフロー可視化、履歴検索を統合する。

最初の相談としては、MVPは「ワークフロー全体図 + 現在位置 + 根拠ログ」までに絞るのがよいと思います。LLM対話ペインは強力ですが、最初から入れるとアプリの責務が広がるので、第2段階が自然です。

```text
ReviewCompass 本体
```

```yaml
  - id: triage

    status: completed
    evidence:
      - path: triage.yaml

  - from: triage
    to: post_write_verification
```

ビューアはこのスナップショットだけを信じる。根拠ファイルへのリンクは出すけれど、状態判定は本体側で済ませる。

**アプリごとの機能軸**

ここも固定の「requirements → design → tasks → implementation」だけでは足りません。開発対象によって軸が変わるので、ビューアは工程を固定せず、**任意のノード/エッジ/レーンを描ける汎用ビューア**にするのがよいです。

たとえば ReviewCompass なら：

```text
Intent → Requirements → Design → Tasks → Implementation → Review → Triage
```

Webアプリ開発なら：

```text
Spec → UI Design → API → Implementation → Browser QA → Release
```

データ分析なら：

```text
Question → Dataset → Cleaning → Analysis → Validation → Report
```

LLM評価なら：

```text
Prompt → Model Runs → Parsing → Metrics → Judgment → Archive
```

つまりビューアに必要なのは「ReviewCompass専用の段階名」ではなく、

- ノード
- エッジ
- レーン
- 状態
- 根拠
- 次アクション
- 表示優先度

を受け取って描く機能です。

**接続モデル**

私はこう分けるのがよいと思います。

```text
ReviewCompass Core
  - 状態判定
  - ワークフロー定義
  - 根拠ファイル収集
  - current_node 判定
  - snapshot export

Viewer
  - snapshot 読み込み
  - グラフ描画
  - 詳細ペイン
  - ログ/根拠表示
  - 差分表示
```

これなら、別アプリでも「同じ snapshot schema を吐けば同じビューアで見られる」状態になります。

**最初に決めるべき契約**

MVPの前に、まず `workflow-state.schema.json` を定義するのがよさそうです。TDDでやるなら最初のテスト対象はUIではなく、

- ReviewCompass が正しい snapshot を出力できる
- Viewer が snapshot schema に従って描画できる
- 不明なノード状態を壊さず表示できる
- アプリ固有のレーン/軸を固定名なしで扱える

あたりになります。

結論として、静的ビューアのMVPは「ReviewCompassの内部ファイルビューア」ではなく、**ReviewCompass Core が出力する workflow-state を読む汎用グラフィックビューア**として作るのがよいです。

```text
規定ルート:

実際:
A done
B done
C partially done
D manually touched
E not started

Navigator:
現在位置: C
例外操作: D に痕跡あり
次に必要: C の完了判定を満たす
復帰先: C → D → E
```

## アシスタント（2026-06-05T11:54:39.044Z）

基本は、**git tree 自体も Navigator の状態入力にする**のがよいと思います。

`BTW Track` に入ったとき、git tree はこう扱う。

```text
main workflow の現在位置
  +
git diff にある対象変更
  +
post-write manifest の検証状態
= 現在位置
```

つまり、BTW中は「正規ワークフローを進めている」のではなく、**dirty tree を解消する補助系統にいる**と見なす。

おすすめは段階で分けることです。

**小さい BTW**
同じ worktree のまま扱う。

```text
変更: docs/notes/foo.md
状態: post-write-verification pending
許可: 対象ファイルの検証、manifest 作成、必要最小限の字句修正
禁止: unrelated file の編集、実装変更、別仕様変更
```

この方式は既存の post-write-verification と相性がいいです。未コミット変更がそのまま検証対象なので、sha256 と manifest で機械判定できます。

**大きい BTW**
別 branch か別 worktree に切る。

```text
main branch/worktree:
  正規ワークフローの作業位置を保持

btw branch/worktree:
  例外修正、検証、必要なら commit
```

ただし、最初のMVPでは branch/worktree 自動分離までやらなくてよさそうです。まずは同一 worktree 上で、

- `git diff --name-only`
- post-write 対象ファイルか
- manifest が現在 sha256 を覆うか
- 非対象変更が混ざっていないか

を見て、Navigator が状態表示する。

表示はこういう感じがよいです。

```text
Active Track: BTW / Post-write Verification
Git Tree: dirty
Target Changes:
- docs/notes/foo.md

Blocked Changes:
- tools/bar.py

Return To:
workflow-management / implementation / review-wave
```

重要なのは、BTW中の dirty tree を「混乱」ではなく、**一時的な状態として正規化する**ことです。

なので原則は：

```text
同一 worktree:
  小規模・検証対象が明確・manifest で閉じられる BTW

別 worktree/branch:
  複数ファイル大規模変更・実装変更を含む・正規ワークフローと並行したい BTW
```

Navigator の schema には `git_state` を持たせたいです。

```yaml
git_state:
  tree_status: dirty
  active_track: post_write_verification
  target_files:
    - docs/notes/foo.md
  non_target_changes: []
  manifest_status: pending
  return_to:
    outer: implementation
    inner: review_wave
```

これで「git tree が汚れているから怖い」ではなく、「汚れている理由・検証対象・復帰先が機械的にわかる」状態になります。
