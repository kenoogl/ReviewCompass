`reopen-status` が存在すると誤認して手戻りが発生した。reopen 状態確認は `next --json` と in-progress YAML 確認が正規経路であり、存在しない補助コマンド名を推測して実行しないよう、reopen 操作前に `--help` または command registry を確認する preflight を追加する。



`check-workflow-action.py next` が `--file` を受け取ると誤認し、コマンド指定の手戻りが発生した。reopen / next / triage などの workflow CLI を使う前に、対象サブコマンドの存在・引数・前提ファイルを `--help` または機械可読 command registry で確認する preflight が必要。



既存の「手戻り・機械処理化候補を集約するメモ」に追記するのがよいです。

書く内容は、単に「implementation review target に diff がなかった」ではなく、原因を一段上げてこう整理します。

**メモ項目案**

- 件名：前例模倣による implementation review target 不備
- 発生したこと：implementation triad-review の `review-target.md` を作る際、requirements/design/tasks の前例形式を模倣し、実装 diff や主要コード抜粋を含めなかった。
- 直接原因：文書レビュー用の review-target 形式を、implementation review にそのまま横展開した。
- 本質原因：前例を使う前に、正本・phase・artifact 種別・reviewer が必要とする一次情報を照合しなかった。
- 影響：review-run は実行できたが、reviewer から「実装を検証できない」と指摘され、review-run や review target の作り直しが必要になった。
- 対策候補：
  - 前例適用前チェックを追加する。
  - review-run 作成前に `phase=implementation` なら diff / code excerpt / test diff の有無を検査する。
  - 「前例と同じ形か」ではなく「今回の reviewer が検証可能か」をチェック項目にする。
- 分類：review-run / bundle 作成前チェック、前例適用チェック、手戻り削減候補。

平たく言うと、**「diff が無かった」ではなく、「前例を正本照合なしに模倣した」ことを主原因として記録する**のが大事です。



## post-write pending 中の別作業混在による workflow 混線

### 発生したこと

既存の post-write-verification pending が残っている状態で、別件の実装変更を同じ worktree に追加したため、`next` / guard が「post-write pending 中にコード変更が混ざっている」と判定して停止した。

今回の具体例では、smell inventory 系の post-write-verification 後処理が残っているところに、commit execution delegation 実装変更が同居した。

### 性質

今回の直接原因は、過渡的な staged / dirty 状態の同居である。

ただし問題の型としては定常的に起こりうる。ReviewCompass の運用では、review-run、post-write-verification、reopen、commit approval などの未完了手続き中に、別の不具合や改善点を発見して同じ worktree で修正を始める場<truncated omitted_approx_tokens="53" />の分離が手作業になり、手戻り・説明コスト・トークン消費が増える。
- 無理に進めると、post-write 対象外の変更が同じ流れに混入する危険がある。

### 本質原因

未完了手続きがある状態で、新しい作業差分を同じ worktree に作る前の機械的な分離チェックが弱い。

「pending 手続きの対象差分」と「新しく始める作業差分」が同居してよいかを、作業開始前に判定できていない。

### 対策候補

- 作業開始前チェックで、pending 手続きと staged / dirty 差分の種類を照合する。
- post-write pending 中に対象外ファイルを変更しようとした場合、開始前に停止して分離案を出す。
- 分離案は以下を機械的に提示する：
  - 先に pending 手続きを完了する
  - 別 worktree / 別スレッドへ分ける
  - 現在作業を保留メモ化する
  - 明示承認を得て side-track として記録する
- `next` が混在を検出したとき、単に止めるだけでなく「どの既存 pending と、どの新規差分が衝突しているか」を表示する。

### 分類

- workflow 混線防止
- post-write / review-run / reopen 作業分離
- 作業開始前チェック
- 手戻り削減候補



これは構造がはっきりしています。

- `completed_gates` に gate が並ぶ
- `pending_gates` から外れた gate も含め、完了扱いの gate がある
- それぞれに対応する `downstream_impact_decisions[].gate` が必要
- なければ記録漏れ

なので、機械チェックしやすいです。

具体的には、reopen 完了前チェックで次を見ればよいです。

1. `completed_gates` を読む
2. `drafting_completed_gates` は別扱いにする
3. `requirements/design/tasks/implementation` の `triad-review/review-wave/alignment/approval` gate を対象にする
4. 各 gate について、`downstream_impact_decisions[].gate` に同じ値があるか確認する
5. なければ `DEVIATION` で止める

さらに、補助的に自動生成もできます。

- `reopen-advance-gate` を使って gate を完了させたとき、同時に `downstream_impact_decisions` へ必ず1件追加する
- `completed_gates` へ手で追加する経路を禁止する
- 既存ファイルの修復用に、欠落 gate 一覧を出す lint を作る

ただし、**判断理由そのものは完全自動では作らない方がよい**です。
機械ができるのは「対応レコードがあるか」「必須フィールドがあるか」「evidence が空でないか」までです。
`rationale` の中身が本当に妥当かは、人または review/proxy が見る必要があります。

つまり：

- **記録漏れ検出**：機械処理できる
- **必須フィールド検査**：機械処理できる
- **どの gate が足りないかの提示**：機械処理できる
- **rationale の意味的妥当性**：完全自動は難しい

対応候補としては、`check-workflow-action.py commit` または `reopen finalize` 系の検査に、**completed_gates と downstream_impact_decisions の対応検査**を入れるのが自然です。







現状の弱点は、この順序が「手順として分かっている」だけで、**実行側が並列化しようとした時点で止める機械ガードがない**ことです。

対策としては、メモに以下を入れるのがよいです。

- commit approval chain は順序依存処理として分類する
- `prepare → record → delegate-execution → guarded commit` は直列専用
- ツールまたは wrapper で一括実行し、途中成果物の存在確認と digest 検証を挟む
- `multi_tool_use.parallel` などの並列実行対象から明示的に除外する

平たく言うと、**ルールはあるが、LLM が間違えないようにする実行前ガードが足りない**、という状態です。

## post-write pending 中の別作業混在による workflow 混線

### 発生したこと

既存の post-write-verification pending が残っている状態で、別件の実装変更を同じ worktree に追加したため、`next` / guard が「post-write pending 中にコード変更が混ざっている」と判定して停止した。

今回の具体例では、smell inventory 系の post-write-verification 後処理が残っているところに、commit execution delegation 実装変更が同居した。

### 性質

今回の直接原因は、過渡的な staged / dirty 状態の同居である。

ただし問題の型としては定常的に起こりうる。ReviewCompass の運用では、review-run、post-write-verification、reopen、commit approval などの未完了手続き中に、別の不具合や改善点を発見して同じ worktree で修正を始める場<truncated omitted_approx_tokens="53" />の分離が手作業になり、手戻り・説明コスト・トークン消費が増える。
- 無理に進めると、post-write 対象外の変更が同じ流れに混入する危険がある。

### 本質原因

未完了手続きがある状態で、新しい作業差分を同じ worktree に作る前の機械的な分離チェックが弱い。

「pending 手続きの対象差分」と「新しく始める作業差分」が同居してよいかを、作業開始前に判定できていない。

### 対策候補

- 作業開始前チェックで、pending 手続きと staged / dirty 差分の種類を照合する。
- post-write pending 中に対象外ファイルを変更しようとした場合、開始前に停止して分離案を出す。
- 分離案は以下を機械的に提示する：
  - 先に pending 手続きを完了する
  - 別 worktree / 別スレッドへ分ける
  - 現在作業を保留メモ化する
  - 明示承認を得て side-track として記録する
- `next` が混在を検出したとき、単に止めるだけでなく「どの既存 pending と、どの新規差分が衝突しているか」を表示する。

### 分類

- workflow 混線防止
- post-write / review-run / reopen 作業分離
- 作業開始前チェック
- 手戻り削減候補



これは構造がはっきりしています。

- `completed_gates` に gate が並ぶ
- `pending_gates` から外れた gate も含め、完了扱いの gate がある
- それぞれに対応する `downstream_impact_decisions[].gate` が必要
- なければ記録漏れ

なので、機械チェックしやすいです。

具体的には、reopen 完了前チェックで次を見ればよいです。

1. `completed_gates` を読む
2. `drafting_completed_gates` は別扱いにする
3. `requirements/design/tasks/implementation` の `triad-review/review-wave/alignment/approval` gate を対象にする
4. 各 gate について、`downstream_impact_decisions[].gate` に同じ値があるか確認する
5. なければ `DEVIATION` で止める

さらに、補助的に自動生成もできます。

- `reopen-advance-gate` を使って gate を完了させたとき、同時に `downstream_impact_decisions` へ必ず1件追加する
- `completed_gates` へ手で追加する経路を禁止する
- 既存ファイルの修復用に、欠落 gate 一覧を出す lint を作る

ただし、**判断理由そのものは完全自動では作らない方がよい**です。
機械ができるのは「対応レコードがあるか」「必須フィールドがあるか」「evidence が空でないか」までです。
`rationale` の中身が本当に妥当かは、人または review/proxy が見る必要があります。

つまり：

- **記録漏れ検出**：機械処理できる
- **必須フィールド検査**：機械処理できる
- **どの gate が足りないかの提示**：機械処理できる
- **rationale の意味的妥当性**：完全自動は難しい

対応候補としては、`check-workflow-action.py commit` または `reopen finalize` 系の検査に、**completed_gates と downstream_impact_decisions の対応検査**を入れるのが自然です。







現状の弱点は、この順序が「手順として分かっている」だけで、**実行側が並列化しようとした時点で止める機械ガードがない**ことです。

対策としては、メモに以下を入れるのがよいです。

- commit approval chain は順序依存処理として分類する
- `prepare → record → delegate-execution → guarded commit` は直列専用
- ツールまたは wrapper で一括実行し、途中成果物の存在確認と digest 検証を挟む
- `multi_tool_use.parallel` などの並列実行対象から明示的に除外する

平たく言うと、**ルールはあるが、LLM が間違えないようにする実行前ガードが足りない**、という状態です。

## post-write pending 中の別作業混在による workflow 混線

### 発生したこと

既存の post-write-verification pending が残っている状態で、別件の実装変更を同じ worktree に追加したため、`next` / guard が「post-write pending 中にコード変更が混ざっている」と判定して停止した。

今回の具体例では、smell inventory 系の post-write-verification 後処理が残っているところに、commit execution delegation 実装変更が同居した。

### 性質

今回の直接原因は、過渡的な staged / dirty 状態の同居である。

ただし問題の型としては定常的に起こりうる。ReviewCompass の運用では、review-run、post-write-verification、reopen、commit approval などの未完了手続き中に、別の不具合や改善点を発見して同じ worktree で修正を始める場<truncated omitted_approx_tokens="53" />の分離が手作業になり、手戻り・説明コスト・トークン消費が増える。
- 無理に進めると、post-write 対象外の変更が同じ流れに混入する危険がある。

### 本質原因

未完了手続きがある状態で、新しい作業差分を同じ worktree に作る前の機械的な分離チェックが弱い。

「pending 手続きの対象差分」と「新しく始める作業差分」が同居してよいかを、作業開始前に判定できていない。

### 対策候補

- 作業開始前チェックで、pending 手続きと staged / dirty 差分の種類を照合する。
- post-write pending 中に対象外ファイルを変更しようとした場合、開始前に停止して分離案を出す。
- 分離案は以下を機械的に提示する：
  - 先に pending 手続きを完了する
  - 別 worktree / 別スレッドへ分ける
  - 現在作業を保留メモ化する
  - 明示承認を得て side-track として記録する
- `next` が混在を検出したとき、単に止めるだけでなく「どの既存 pending と、どの新規差分が衝突しているか」を表示する。

### 分類

- workflow 混線防止
- post-write / review-run / reopen 作業分離
- 作業開始前チェック
- 手戻り削減候補



これは構造がはっきりしています。

- `completed_gates` に gate が並ぶ
- `pending_gates` から外れた gate も含め、完了扱いの gate がある
- それぞれに対応する `downstream_impact_decisions[].gate` が必要
- なければ記録漏れ

なので、機械チェックしやすいです。

具体的には、reopen 完了前チェックで次を見ればよいです。

1. `completed_gates` を読む
2. `drafting_completed_gates` は別扱いにする
3. `requirements/design/tasks/implementation` の `triad-review/review-wave/alignment/approval` gate を対象にする
4. 各 gate について、`downstream_impact_decisions[].gate` に同じ値があるか確認する
5. なければ `DEVIATION` で止める

さらに、補助的に自動生成もできます。

- `reopen-advance-gate` を使って gate を完了させたとき、同時に `downstream_impact_decisions` へ必ず1件追加する
- `completed_gates` へ手で追加する経路を禁止する
- 既存ファイルの修復用に、欠落 gate 一覧を出す lint を作る

ただし、**判断理由そのものは完全自動では作らない方がよい**です。
機械ができるのは「対応レコードがあるか」「必須フィールドがあるか」「evidence が空でないか」までです。
`rationale` の中身が本当に妥当かは、人または review/proxy が見る必要があります。

つまり：

- **記録漏れ検出**：機械処理できる
- **必須フィールド検査**：機械処理できる
- **どの gate が足りないかの提示**：機械処理できる
- **rationale の意味的妥当性**：完全自動は難しい

対応候補としては、`check-workflow-action.py commit` または `reopen finalize` 系の検査に、**completed_gates と downstream_impact_decisions の対応検査**を入れるのが自然です。







現状の弱点は、この順序が「手順として分かっている」だけで、**実行側が並列化しようとした時点で止める機械ガードがない**ことです。

対策としては、メモに以下を入れるのがよいです。

- commit approval chain は順序依存処理として分類する
- `prepare → record → delegate-execution → guarded commit` は直列専用
- ツールまたは wrapper で一括実行し、途中成果物の存在確認と digest 検証を挟む
- `multi_tool_use.parallel` などの並列実行対象から明示的に除外する

平たく言うと、**ルールはあるが、LLM が間違えないようにする実行前ガードが足りない**、という状態です。
