    target_location: "tools/check-workflow-action.py / commit approval consume path"
    description: "Challenge consumption persistence appears incomplete: the review target states guarded commit consumes the approval record, but challenge consumption may not be durably persisted after successful commit."
    rationale: "Requirement 4 acceptance 6 and the irreversible-operation gate model depend on a nonce challenge being single-use. If only the approval record is consumed while the challenge remains reusable or unmarked on disk after commit, the same nonce-bound authorization can be replayed against the same staged state within TTL. The target document itself flags this as a known concern and does not cite tests proving durable challenge consumption, so implementation approval should not complete without confirming and testing one-time nonce enforcement."

  - severity: WARN
    target_location: "tools/check_workflow_action/commit_approval.py / commit-approval record output schema"
    description: "Approval records are described as always embedding execution_delegation delegated_to=llm by default, which conflates nonce approval capture with delegation to a specific execution mode."
    rationale: "The reviewed requirement is about recording user commit approval through a nonce bound to staged content, independent of controlling LLM/provider/model. Automatically asserting execution delegation to an LLM in every recorded approval weakens that separation of concerns and can make the approval record semantically broader than the user's nonce acknowledgment. This does not necessarily break machine validation, but it creates avoidable ambiguity in the attestation boundary and should be narrowed or explicitly justified."

Without dedicated tests for these cases, there is insufficient evidence that the validator safely rejects malformed challenge/approval structures and that the digest is stable across the full set of git index states required by design/tasks."
