findings:
  - severity: ERROR
    target_location: "tools/check_workflow_action/approval_gate.py: validate_approval_gate_record, allows_target_operation"
    description: "The `decided_by` field is never validated against its allowed value set. The schema defines `decided_by` as an enum of `user`, `human`, `proxy_model`, `llm`, but the implementation contains no check that rejects an out-of-enum or unexpected actor value. An invalid `decided_by` (e.g., `decided_by='llm'`) passes validation without error."
    rationale: "Upstream design (design.md lines 1820-1833) requires that `decided_by=proxy_model` and `decision_scope=human_only` records are not treated as approval. While the proxy/human_only check in `allows_target_operation` catches `proxy_model`, the absence of a decided_by enum validation means the field can carry arbitrary strings. This weakens the actor/source binding required by acceptance criteria (binding actor, source) and may allow unexpected actor values to slip through undetected."

  - severity: ERROR
    target_location: "tools/check_workflow_action/approval_gate.py: allows_target_operation"
    description: "The `consumed` guard only checks for `consumed is True`, but never marks or enforces that a record is consumed after use. The function checks `record.get('consumed') is True` and blocks, but there is no write-path or enforcement mechanism preventing the same unconsumed record from being passed to `allows_target_operation` multiple times. Replay of an approval gate record is not prevented at the validation layer."
    rationale: "Upstream acceptance criteria require that record_human_decision completion is not conflated with target-operation approval, and that non-approved decisions block operations. Single-use consumption enforcement is part of the gate binding model (schema `consumed: boolean`). Without any replay-prevention check or documentation of enforcement responsibility, the current implementation is ambiguous about whether the consumed field is ever set to true, leaving an unsupported gap that could allow the same approval record to be reused."

  - severity: WARN
    target_location: "tools/check_workflow_action/approval_gate.py: HUMAN_ONLY_REQUIRED_ACTIONS"
    description: "The `HUMAN_ONLY_REQUIRED_ACTIONS` set does not include `commit`, `push`, or `spec_json_update` as required actions, although upstream design (requirements.md lines 254-255, design.md lines 1820-1833) lists `commit`, `push`, `spec.json update`, `phase approval`, and `reopen finalize` as the initial human-only decision set. `finalize_reopen` is present, but `commit_stop_point` is the only commit-adjacent entry and neither `push` nor `spec_json_update` appear."
    rationale: "If the operation registry uses required_action identifiers such as `push` or `update_spec_json` that do not appear in HUMAN_ONLY_REQUIRED_ACTIONS, `derived_decision_scope` will fall through to lower-priority checks. Depending on the operation contract shape, a push or spec.json update could be classified as `proxy_allowed` rather than `human_only`. This is a coverage gap relative to the upstream human-only boundary definition."

  - severity: WARN
    target_location: "tests/workflow-management/test_approval_gate.py: test_binding_kind_none_is_allowed_only_for_wait_only_operations"
    description: "The wait-only `binding_kind=none` positive test uses `decision_scope='advisory_only'`, but the `_allows_binding_none` implementation requires `approval_required=False`, `effect_kind='read'`, and `phase_boundary='none'`. The test does not verify that a wait-only operation with `decision_scope='proxy_allowed'` or `decision_scope='human_only'` and `binding_kind=none` is also blocked. Coverage for the boundary between `advisory_only` and other scopes with `binding_kind=none` is absent."
    rationale: "Upstream acceptance criteria require focused tests covering the proxy/human boundary and binding/digest checks. The existing test exercises one valid path but does not exercise the reject path where a non-advisory scope is combined with `binding_kind=none` for a wait-only operation, leaving a gap in the required test coverage."

  - severity: WARN
    target_location: "tools/check_workflow_action/approval_gate.py: validate_approval_gate_record"
    description: "The `source_ref` and `source_digest` fields are only checked for presence (via REQUIRED_RECORD_FIELDS), but their content is never validated. There is no check that `source_digest` is a non-empty string in the expected hash format, and no check that `source_ref` is non-empty beyond its inclusion in the required-fields list."
    rationale: "Upstream design (requirements.md lines 240-246) binds each decision to its source (`source_ref`, `source_digest`). Accepting blank or structurally invalid source fields would weaken the source-binding guarantee. The schema enforces `minLength: 1` on both fields, but the Python validation does not apply this constraint."

  - severity: INFO
    target_location: "tests/workflow-management/test_approval_gate.py: test cases overall"
    description: "There are no test cases covering `decided_by='llm'` attempting to satisfy a `human_only` gate, nor tests for `decided_by='human'` (as opposed to `'user'`) combined with `decision_scope='human_only'`. The `decided_by` enum includes both `user` and `human` as separate values, but only `user` is exercised in the baseline valid record."
    rationale: "Minor coverage gap against the full actor/scope matrix required by upstream acceptance criteria. Does not block the primary approval-gate behaviors but leaves untested combinations that could reveal edge cases."
