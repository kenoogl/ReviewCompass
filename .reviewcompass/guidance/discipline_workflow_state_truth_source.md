workflow_state（spec.json）を状態判定の真実源として扱い、過去確定事項は出典付きのみ信頼する。

**読む側の規律：**

- 状態判定時は必ず workflow_state を実際に Read で読む
- 要約や作業メモの記述を根拠にしない（正本と照合）

**書く側の規律：**

- 要約値（current_phase 相当）を spec.json に書かない
- 応答で単独断定しない
- 状態表現には workflow_state の根拠を必ず併記

**作業開始時の引継ぎ：**

- 引継ぎメモや作業メモの過去確定事項は出典付きのみ信頼
- 出典なし項目は正本、ログ、または人の確認で再検証
- 未確認の要約を次の作業判断へ伝播させない

**Why:** 状態判断を要約や記憶に依存させると、古い状態や未確認の作業メモが次の判断へ混入する。workflow_state と出典付き記録を優先することで、作業順序と承認状態を再確認可能にする。

**How to apply:**

- 作業開始時の状態確認：spec.json の workflow_state を読む。要約資料に頼らない
- 「○○段が approved である」と書く前：その状態の出典（ファイル行）を併記
- 引継ぎメモに書かれた確定事項を踏襲する前：出典の有無を確認し、なければ正本、ログ、または人の確認で再検証
- spec-set などの機械判定がある場合は、その判定結果を優先し、応答では根拠を併記する
