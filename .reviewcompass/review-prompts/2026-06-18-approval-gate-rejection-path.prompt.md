# 設計深掘りレビュー：承認ゲートの拒否・保留経路

## 設計の前提

ReviewCompass の承認ゲートは現在、§5.1 で次のように定義されている。

```
wait_for_human_decision（LLM が判断材料を提示し、人間が判断を下す）
  → record_human_decision（人間の判断内容を構造化ファイルに書き出す）
    → 対象の不可逆操作を実行する
```

## 問題の状況

この定義は「人間が承認した場合」の経路しか記述していない。

実際の開発では、人間が次のいずれかを行う可能性がある。

- **承認（approve）**：対象の不可逆操作を実行してよいと判断する
- **拒否（reject）**：対象の不可逆操作を実行しないと判断する
- **保留（defer）**：今は判断できない、または追加情報が必要と判断する
- **条件付き承認（conditional approve）**：一部を修正したうえで実行してよいと判断する

現状の設計では、承認以外の結果が出たとき `next --json`（次に実行すべき操作を返す仕組み）が何を返すべきかが未定義である。

## レビューで答えてほしい問い

1. 承認ゲートにおいて、人間が「承認以外」の判断を下した場合、
   システムはその後どのような状態になるべきか。
   具体的なシナリオで示すこと（例：reopen の approval gate で人間が拒否した場合）。

2. 拒否・保留・条件付き承認のそれぞれについて、
   `next --json` は次に何を返すべきか。
   それぞれの返値の違いと、その違いが必要な理由を示すこと。

3. `record_human_decision` は承認結果だけでなく、拒否・保留も記録すべきか。
   記録すべきだとすれば、承認の場合と同じコマンドで処理できるか、
   別の仕組みが必要か。

4. 承認ゲートの定義（§5.1）をどう書き直せば、承認・拒否・保留の全経路を表現できるか。
   修正案を具体的に示すこと。

## 出力形式

YAML 形式で次の構造で出力すること。

```yaml
findings:
  - severity: ERROR / WARN / INFO
    question_id: "1" / "2" / "3" / "4"
    description: |
      （分析内容）
    rationale: |
      （根拠）
```
